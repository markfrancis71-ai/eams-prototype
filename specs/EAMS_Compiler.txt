1. Overview
The EAMS Compiler is the pipeline that transforms DSL source files in Git into the materialised architecture graph. It is the single authoritative write path in the EAMS ecosystem — every entity record in OpenSearch, every component in Ardoq, every compliance finding, and every diff report originates from a compiler run. No other component writes architecture data.
The compiler runs in two modes: CI mode (triggered by a Git push via GitHub Actions) and local mode (invoked directly by an architect via the eams CLI). Both modes execute the same pipeline stages. They differ only in invocation, output destination, and concurrency behaviour.
1.1 Design Decisions
Three-tier error classification: Errors are classified as hard (block the affected entity and its dependents), soft (generate findings but never block), or advisory (completeness warnings surfaced to the architect). This maximises the number of entities that succeed on every compile while being strict where correctness matters.
Git diff plus dependency graph for incremental compiles: The reprocess set is the union of files changed in the commit and any files that declare relationships to changed entities. This closes the referential integrity gap in git-diff-only incremental compiles without requiring a full recompile.
Queued concurrency: A second compile trigger while a compile is in progress waits in a DynamoDB-backed queue. Compiles on the same branch run sequentially. This is simpler than branch-isolated parallelism and correct for v1; branch-isolated concurrency is a future optimisation.
Both human-readable and JSON output: The CLI produces human-readable output with file paths and line numbers by default. The --json flag produces structured JSON for CI pipeline integration. Both formats carry identical information.
Idempotent pipeline stages: Every pipeline stage is designed to be safely re-run. If a stage fails partway through, re-running the compile from scratch produces the same result as completing the original run. This means the compiler never needs a rollback mechanism — re-run is the recovery action.
Compile record as the audit trail: Every compile run produces a compile record in DynamoDB capturing the trigger, branch, git SHA, stage outcomes, entity counts, findings, and duration. This is the authoritative audit trail for the architecture graph's evolution.

2. Pipeline Stages
The compiler pipeline has eight stages, executed in sequence. Stages 1–4 are read-only (no external writes). Stages 5–8 write to external systems. This separation means that validation failures in stages 1–4 abort the compile before any external state is modified.
2.1 Stage 1 — Source Loading
Git operations
In CI mode the repository is available as a fresh clone in the GitHub Actions runner. In local mode, eams compile runs against the working directory. The compiler reads the current branch name, current HEAD SHA, and the SHA of the last successful compile for this branch from the DynamoDB compile record table.
2.2 Stage 2 — Schema Validation
Error classification
Every validation error produced in stage 2 is classified into one of three tiers. The classification determines whether the entity proceeds through the pipeline.
2.3 Stage 3 — Referential Integrity
Referential integrity validation ensures every ID referenced in a DSL file corresponds to a real entity. References that cross domain boundaries are validated against the full entity set loaded from the previous compile's OpenSearch index (for unchanged entities) and the current reprocess set (for changed entities).
2.4 Stage 4 — Standards Compliance Check
The compliance check evaluates each entity against its applicable standards. Standards define mandatory fields, field value constraints, and relationship requirements. The compliance engine is distinct from schema validation: schema validation checks the DSL file format; compliance checks whether the entity's declared attributes satisfy the organisation's architectural standards.
2.5 Stage 5 — Diff Computation
The diff compares the current compile's entity set against the previous compile's entity set for the same branch. It produces counts and details of added, modified, and removed entities, which are written to the compile record and surfaced in the Architecture API /diff/{branch} endpoint.
3. Error Handling and Partial Compiles
The compiler distinguishes four compile outcomes. Each has defined behaviour for what is written to external systems and what is reported to the architect.
3.1 Blocked Entity Propagation
When an entity is blocked by a hard error, the compiler must also block any entity that declares a relationship to the blocked entity. This prevents dangling relationships in the graph — an entity that references a non-existent or invalid target would produce a broken relationship record in Ardoq and OpenSearch.
4. Concurrency and Compile Locking
The compiler uses a DynamoDB-backed compile lock to prevent concurrent compiles on the same branch from interleaving their writes to OpenSearch and Ardoq. A second compile trigger that arrives while a compile is already in progress waits in a queue until the lock is released.

5. Branch Compile Semantics
Branch compiles write to branch-isolated targets rather than the main architecture graph. The targets differ by branch intent, as established in the Ardoq Integration Layer specification.
5.1 Branch Compile Targets
5.2 Branch Compile Lifecycle
First compile of a branch: compiler detects no existing branch targets, creates OpenSearch branch index and Ardoq target (Scenario or workspace copy). Writes full entity set.
Subsequent branch compiles: incremental compile against branch-specific OpenSearch index. Ardoq diff-aware writes to branch target.
Branch review: architect requests review via eams review submit. Compile report and diff are attached to the review record.
Merge to main: post-merge GitHub Actions trigger runs a main branch compile. Branch index and Ardoq branch target are marked for cleanup but not immediately deleted.
Branch cleanup: 30 days after merge, a nightly cleanup job deletes the branch OpenSearch index and Ardoq branch target. Compile records are retained indefinitely.
5.3 Branch Conflict Detection
A branch conflict occurs when both the branch and main have modified the same entity since the branch was created. The compiler detects conflicts during the diff computation stage by comparing the branch entity's base SHA (the main branch SHA at branch creation time) against the current main branch SHA for the same entity.

9. Performance
9.1 Compile Duration Targets
9.2 Performance Architecture
Stages 1–4 are CPU-only: No external network calls in validation stages. All validation runs in Lambda memory against loaded DSL files and in-memory indexes. This is the fastest part of the pipeline.
Stages 6–7 are the bottleneck: OpenSearch and Ardoq writes dominate compile duration. Both are batched (50 entities per batch) and parallelised within a stage (up to 5 concurrent batches). Cross-stage parallelism (6 and 7 simultaneously) is off by default in v1 but available via the parallel_stages config flag.
Ardoq rate limit management: The Ardoq Integration Layer's write throttle (200ms inter-batch delay) is the primary Ardoq rate limit control. On large compiles, this is the limiting factor. The throttle is configurable; increasing it risks 429 responses from Ardoq.
OpenSearch bulk indexing: Stage 6 uses the OpenSearch Bulk API (POST /_bulk) rather than individual document writes. Batch size is 50 documents, approximately 200KB per batch. Indexing 200 entities takes approximately 4–8 seconds under normal conditions.

1 | Source Loading | Read | Yes | Clone or fetch the Git repository. Identify changed files via git diff against last compile SHA. Expand reprocess set via dependency graph walk.
2 | Schema Validation | Read | Hard errors only | Parse each DSL file. Validate YAML syntax, required fields, enum values, field format constraints. Classify errors.
3 | Referential Integrity | Read | Hard errors only | Verify all relationship targets exist. Verify all referenced standards, ADRs, and capabilities exist. Build in-memory entity graph.
4 | Standards Compliance Check | Read | Never | Evaluate each entity against its applicable standards. Generate compliance findings. No entity is blocked by a compliance finding.
5 | Diff Computation | Read + Write | Yes | Compare current entity set against previous compile record. Compute added/modified/removed counts. Write diff to compile record.
6 | OpenSearch Indexing | Write | Soft | Write validated entity and relationship records to OpenSearch. Diff-aware: only write changed entities in incremental mode.
7 | Ardoq Integration | Write | Soft | Write entities and relationships to Ardoq via the Integration Layer. ID Registry updates. Diff-aware.
8 | Compile Record Finalisation | Write | Yes | Write final compile record to DynamoDB. Publish compile completion event. Trigger Workbench entityUpdated subscription. Refresh prompt cache if prompt files changed.
Stage failure semantics
Stages 1–3 failures abort the compile immediately — external systems are never written to. Stage 4 never blocks.
Stage 5 failure (diff computation) blocks because it would leave the compile record in an inconsistent state.
Stages 6–7 failures are soft: a partial write is recorded in the compile record. The compile is marked partial-success. Affected entities are retried on the next compile. Stage 8 always runs even after stage 6–7 partial failures — the compile record must be finalised.
Source loading — incremental reprocess set computation
# Step 1: Get last successful compile SHA for this branch
const lastSha = await getLastCompileSha(branch)  // DynamoDB lookup

# Step 2: Git diff to find changed DSL files
const changedFiles = await gitDiff(lastSha, currentSha, '**/*.yaml')

# Step 3: Load last compile's relationship index (in-memory)
#          Maps entity_id -> [entity_ids that declare relationships TO this entity]
const reverseRelIndex = await loadReverseRelIndex(branch, lastSha)

# Step 4: Expand reprocess set
#          For each changed entity, add all files that reference it
const reprocessSet = new Set(changedFiles)
for (const file of changedFiles) {
const entityId = entityIdFromFilePath(file)
const dependents = reverseRelIndex.get(entityId) ?? []
for (const dep of dependents) reprocessSet.add(entityFileFromId(dep))
}

# Step 5: If no last compile exists (first run), reprocess set = all DSL files
if (!lastSha) reprocessSet = await glob('**/*.yaml', { cwd: repoRoot })

return { changedFiles, reprocessSet, unchangedFiles: allFiles - reprocessSet }

First compile and force-full-sync
When no previous compile record exists for a branch, the compiler treats all DSL files as changed and processes everything. This is the bootstrap path.
The --force flag on eams compile bypasses incremental logic entirely and reprocesses all files regardless of git diff. Used after a schema migration, a standards definition change that affects all entities, or when the compile record is suspected to be stale.

Schema validation — error classification pseudocode
function classifyError(errorType, field, entityKind) {
const HARD_ERROR_TYPES = [
'YAML_PARSE_FAILURE', 'MISSING_REQUIRED_FIELD', 'INVALID_ENUM_VALUE',
'MALFORMED_EAMS_ID', 'INVALID_VERSION_FORMAT', 'DUPLICATE_ID'
]
const SOFT_ERROR_TYPES = [
'STANDARDS_VIOLATION', 'NFR_CONFLICT', 'MISSING_STANDARDS_FIELD'
]
if (HARD_ERROR_TYPES.includes(errorType)) return 'hard'
if (SOFT_ERROR_TYPES.includes(errorType)) return 'soft'
return 'advisory'
}

function shouldBlockEntity(entity, errors) {
return errors.some(e => e.tier === 'hard' && e.entityId === entity.id)
}

function shouldBlockDependent(entity, blockedSet, reverseRelIndex) {
# Block entity if it declares a relationship to a blocked entity
const deps = reverseRelIndex.get(entity.id) ?? []
return deps.some(d => blockedSet.has(d))  // Note: relationship *target* blocked, not source
# Actually block the SOURCE entity if its relationship TARGET is blocked
}

Standards compliance check — evaluation loop
async function checkCompliance(entities, standardsCatalog) {
const findings = []

for (const entity of entities) {
const applicableStandards = standardsCatalog
.filter(s => appliesToEntity(s, entity))  // domain, kind, attribute-based applicability

for (const standard of applicableStandards) {
for (const rule of standard.rules) {
const result = evaluateRule(rule, entity)
if (result.violation) {
findings.push({
entityId: entity.id,
standard: standard.id,
rule: rule.id,
severity: rule.severity,  // violation | warning | advisory
finding: result.message,
deterministic: true  // compiler finding vs LLM advisory finding
})
}
}
}
}

return findings
}

Compliance findings never block
Stage 4 findings are always soft or advisory. An entity with ten compliance violations still writes to OpenSearch and Ardoq — its compliance_status field is set to 'violation' and all findings are stored in the compile record and on the entity record.
The rationale: blocking entities on compliance violations would prevent architects from seeing non-compliant architecture in Ardoq, making it harder to assess and remediate. Visibility is more valuable than enforcement at the write layer.
Exception: a standard can declare severity: blocking on a specific rule. This is rare and requires explicit platform engineer approval to add to a standard definition.
Diff computation — change detection
# An entity is ADDED if its eams_id does not appear in the previous compile record
# An entity is REMOVED if it appeared in the previous compile record but not in the current set
# An entity is MODIFIED if its content_hash differs from the previous compile record
# An entity is UNCHANGED if its content_hash matches and it was not in the reprocess set

function computeContentHash(entity) {
# SHA-256 over canonical JSON serialisation
# Excludes: compiled_at, git_sha (these change every compile)
# Includes: all business fields, relationships, status, standards references
return sha256(JSON.stringify(canonicalise(entity)))
}

function computeDiff(currentEntities, previousCompileRecord) {
const prev = new Map(previousCompileRecord.entities.map(e => [e.id, e.content_hash]))
const curr = new Map(currentEntities.map(e => [e.id, e.content_hash]))

return {
added:    [...curr.keys()].filter(id => !prev.has(id)),
removed:  [...prev.keys()].filter(id => !curr.has(id)),
modified: [...curr.keys()].filter(id => prev.has(id) && prev.get(id) !== curr.get(id)),
unchanged:[...curr.keys()].filter(id => prev.has(id) && prev.get(id) === curr.get(id))
}
}

Outcome | Condition | External Writes | Compile Record Status
Success | All entities pass stage 2–3 validation. Stages 6–7 complete without write failures. | All entities written to OpenSearch and Ardoq. | success
Partial success | One or more entities fail hard validation (blocked). Remaining entities pass. Stages 6–7 complete for unblocked entities. | Unblocked entities written. Blocked entities not written — previous records unchanged. | partial_success
Write failure | Stages 6–7 fail for one or more entities due to OpenSearch/Ardoq errors (after retries). | Successfully written entities updated. Failed entities not updated — previous records unchanged. | partial_success with write_failures list
Compile failure | Stage 1, 2 (catastrophic), 3 (catastrophic), or 5 failure. No DSL files parseable, or compile lock acquisition failed. | Nothing written. External state unchanged. | failed
Blocked entity propagation — breadth-first expansion
function propagateBlocks(initialBlocked, allEntities) {
const blocked = new Set(initialBlocked)
const queue = [...initialBlocked]

while (queue.length > 0) {
const entityId = queue.shift()
# Find all entities that declare a relationship TO this blocked entity
const dependents = allEntities.filter(e =>
e.relationships?.some(r => r.to === entityId)
)
for (const dep of dependents) {
if (!blocked.has(dep.id)) {
blocked.add(dep.id)
queue.push(dep.id)  // Propagate transitively
recordBlockCause(dep.id, entityId, 'DEPENDENT_ON_BLOCKED_ENTITY')
}
}
}

return blocked  // Full set including transitively blocked entities
}

Transitive blocking can have wide impact
If a widely-referenced entity (e.g. a shared infrastructure entity or a core system) fails validation, transitive blocking can affect dozens of dependent entities.
The compile report surfaces the block propagation chain clearly: 'container.claims-db was blocked because it declares a relationship to system.policy-admin, which failed validation with error MISSING_REQUIRED_FIELD: eams_id.'
Architects can use eams validate --entity system.policy-admin to check a specific entity locally before pushing, which avoids this scenario entirely.

Event | Behaviour
Compile triggered, no lock held | Acquire lock immediately. Begin compile.
Compile triggered, lock held | Write run ID to lock item's queued_runs list. Lambda returns 202 Accepted. Polling endpoint available at GET /compile/{run_id}/status.
Compile completes successfully | Release lock. If queued_runs is non-empty, dequeue the oldest entry and trigger it via SQS.
Compile fails | Release lock. Queued runs are dequeued and triggered normally — the queue is not cleared on failure.
Lock TTL expires (lock orphaned) | DynamoDB TTL deletes the lock item. Next compile trigger acquires the lock cleanly. Orphan is logged as a CloudWatch alarm.
Queue depth exceeds 5 | Sixth queued compile is rejected with 429 COMPILE_QUEUE_FULL. Caller must retry after current compile completes.
Branch-isolated parallelism — future optimisation
v1 uses a global queue: any compile, on any branch, waits if any other compile is running. This is conservative but correct.
A future optimisation is per-branch locking: compiles on different branches can run in parallel since they write to separate OpenSearch indexes and separate Ardoq workspaces/scenarios. The lock key would become compile-lock#{branch} rather than a global lock.
Per-branch locking is not implemented in v1 because it requires the Ardoq Integration Layer to confirm that branch workspace writes are fully isolated, which needs operational verification before relying on it for concurrent writes.

Branch conflict detection
function detectConflicts(branchEntities, mainEntities, branchBaseSha) {
const conflicts = []

for (const branchEntity of branchEntities) {
const mainEntity = mainEntities.get(branchEntity.id)
if (!mainEntity) continue  // New entity on branch — no conflict

# Conflict: both branch AND main modified the entity since branch was created
const mainChangedSinceBranch = mainEntity.git_sha !== branchBaseSha
const branchChanged = branchEntity.content_hash !== mainEntity.content_hash

if (mainChangedSinceBranch && branchChanged) {
conflicts.push({
entityId: branchEntity.id,
branchVersion: branchEntity.content_hash,
mainVersion: mainEntity.content_hash,
mainLastModified: mainEntity.compiled_at
})
}
}

return conflicts  // Surfaced in compile report; do not block compile
}

Conflicts are reported, not blocked
A branch compile with conflicts succeeds — the branch entity set is written to the branch targets as-is. Conflicts are surfaced in the compile report and in the Architecture API diff endpoint as a conflicts array.
The architect is responsible for resolving conflicts before the branch is merged. The review submission workflow (eams review submit) warns if the branch has unresolved conflicts.

steps:
- name: Checkout

- name: Setup Node
uses: actions/setup-node@v4  with: {node-version: 22}
- name: Install eams CLI
run: npm install -g @company/eams-cli
- name: Configure AWS credentials
uses: aws-actions/configure-aws-credentials@v4
with: {role-to-assume: ${{ vars.EAMS_COMPILER_ROLE_ARN }}}
- name: Run compile
run: eams compile --json > compile-report.json
env: {EAMS_API_URL: ${{ vars.EAMS_API_URL }}}
- name: Upload compile report
uses: actions/upload-artifact@v4
with: {name: compile-report, path: compile-report.json}
- name: Annotate PR with errors
if: github.event_name == 'pull_request'
uses: company/eams-pr-annotator@v1
with: {report: compile-report.json}

Incremental — small change | 1–5 files changed, < 20 in reprocess set | < 30s | < 90s | Typical developer commit
Incremental — medium change | 5–20 files changed, < 50 in reprocess set | < 2m | < 5m | Feature branch with multiple entity additions
Incremental — large change | 20+ files changed or schema/standards change | < 5m | < 10m | Domain-wide refactor or standard update
Full compile (--force) | All entities across all domains | < 15m | < 30m | Bootstrap, disaster recovery, schema migration
Branch compile — proposal | 1–10 entity changes typical | < 60s | < 3m | Most common branch type
