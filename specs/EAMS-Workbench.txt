1. Overview
The EAMS Architect Workbench is the browser-based interface through which architects, reviewers, and business stakeholders interact with the compiled architecture graph. It provides four integrated workflow surfaces: graph exploration (navigating entities, relationships, and domains), diagram viewing (rendering and browsing viewpoint diagrams with click-through navigation), entity editing (guided form-based editing that commits directly to Git), and advisory review (requesting, reviewing, and acknowledging LLM advisory outputs).
The Workbench is a read surface for most of its users — reviewers and business stakeholders who navigate the architecture without authoring DSL. It progressively reveals authoring and advisory features for users with the Architect role. This separation is reflected in the interface: the core navigation and diagram experience is clean and accessible; editing and advisory controls appear contextually for architects without cluttering the reviewer experience.
1.1 Design Decisions
Split-panel layout: Left panel provides persistent navigation and entity context. Right panel is the active content area, context-switched by navigation actions. This gives architects a stable reference point (left) while the right panel responds to their exploration. The split is resizable; both panels are independently scrollable.
Guided forms for DSL editing: The Workbench provides structured field-input forms for common entity edits. Forms commit to Git via the GitHub API, creating a proper commit with the architect's identity. The DSL file remains the source of truth. Complex edits — new entities, structural changes, relationship additions — are directed to VS Code via a deep-link that opens the file at the correct location.
Navigable diagrams: Clicking a node in a viewpoint diagram opens the corresponding entity detail in the right panel. Hovering an edge shows relationship attributes in a tooltip. Diagrams are rendered from the canonical render graph (SVG) and are never static images. Navigation from diagram to entity and back is seamless.
Progressive disclosure by role: Reader and Reviewer roles see navigation, diagrams, and entity detail. The Architect role additionally sees the advisory panel, the branch and compile workflow, entity edit controls, and the review submission flow. Role is determined at login from Cognito group membership.
Real-time compile status: The Workbench subscribes to the Architecture API's entityUpdated GraphQL subscription. Stale diagrams and entity panels show a staleness indicator with a one-click refresh. The compile status bar shows the current branch's compile state without requiring a page reload.
Branch context persistence: The active branch is persisted in the URL and in localStorage. Navigating to the Workbench always restores the last active branch. The branch switcher is always visible in the top bar.

2. Layout and Navigation
2.1 Application Shell
The application shell is persistent across all views. It contains the top bar and the split-panel workspace. The top bar is always visible; it does not scroll with panel content.
2.2 Left Panel — Navigation Sections
The left panel has five navigation sections, accessible via tab icons at the top of the panel. The active section persists across right-panel context switches.
2.3 Domain Navigation Tree
The Domains section presents the architecture graph as a navigable tree. The tree is lazy-loaded: domain nodes expand to reveal entity kind groups, which expand to reveal individual entities. Entity nodes show a status indicator (active, proposed, deprecated, retired) and a compliance badge if the entity has active violations.
3. Entity Detail Panel
The entity detail panel is the most-visited view in the Workbench. It opens in the right panel when an entity is selected from the navigation tree, clicked in a diagram, or found via search. It presents the entity's complete compiled state, its relationships, its compliance findings, its linked decisions, and its advisory outputs.
3.1 Entity Detail Sections
3.2 Relationship Tab Detail
Each relationship entry in the entity detail panel shows all relationship attributes. Clicking the target entity name opens that entity's detail panel, replacing the current right panel content (with a back navigation breadcrumb). This is the primary graph navigation mechanism.
4. Diagram Panel
The diagram panel renders viewpoint diagrams from the Architecture API's /viewpoints/{id}/resolve endpoint. Diagrams are rendered as interactive SVG in the browser using a lightweight rendering layer over the canonical render graph JSON. The Structurizr DSL, Mermaid, and D2 formats are available as export targets from the diagram panel, but the Workbench renders from the canonical JSON directly rather than embedding a renderer iframe.

4.2 Diagram Node Interactions
4.3 Diagram Rendering Architecture
The Workbench renders diagrams client-side from the canonical render graph JSON returned by the Architecture API. It does not embed Structurizr, Mermaid, or D2 renderers as iframes — these tools are used for export only. Client-side rendering gives the Workbench full control over interactivity, node click handling, and tooltip behaviour.
5. Entity Edit Forms
The entity edit form is the Workbench's primary authoring surface. It provides structured field inputs for common entity edits, validates changes inline before commit, shows a YAML diff preview, and commits to Git via the GitHub API. The form covers the most common edit scenarios — field updates, status changes, relationship additions — without replacing the full expressiveness of the DSL file.

5.2 Edit Form Flow
Architect clicks Edit on an entity detail panel, or double-clicks a diagram node.
Edit form opens as a right-panel overlay (the entity detail remains visible behind a semi-transparent backdrop). The form header shows the entity name and kind.
Form sections correspond to the entity detail sections: Overview, Technology, NFR Profile, Relationships. Sections are collapsible; only sections with editable fields are shown.

Architect clicks Commit. The Workbench calls the GitHub API to create a commit on the current branch. The form closes. The entity detail panel shows a 'Commit pending — waiting for compile' indicator.
On compile completion (via entityUpdated subscription), the entity detail panel refreshes automatically. The 'Commit pending' indicator clears.
5.3 Relationship Edit Form
Adding or editing a relationship has its own sub-form within the entity edit form. The relationship form is the most complex part of the editing surface because relationships have many optional fields and require entity ID autocomplete for the target.
5.4 VS Code Deep-Link
For edit types not supported in the form, the Workbench provides a deep-link that opens the entity's DSL file in VS Code at the correct line. This requires the VS Code EAMS extension to be installed.
6. Advisory Panel
The advisory panel is the Workbench surface for requesting, reviewing, and acknowledging LLM advisory outputs. It is accessible from the left panel navigation (Advisory section) and contextually from entity detail panels and diagram node right-click menus. The review gate logic defined in the LLM Advisory Layer specification is enforced here.
6.1 Requesting an Advisory Output
An architect can request an advisory output from three entry points: the Advisory section in the left panel (domain-scoped requests), the entity detail panel's Advisory section (entity-scoped requests), or the diagram node right-click menu (entity-scoped, specific type).
6.2 Advisory Output Display
Advisory outputs are displayed in the right panel. The display format varies by output type but follows a consistent structure: a header bar with output type, scope entity, generated timestamp, and model tier; a review gate banner for acknowledge-required outputs; the structured output rendered in human-readable form; and the data gaps notice.

6.4 Advisory Queue
The Advisory section in the left panel shows the architect's advisory queue: outputs that have been generated but not yet acknowledged. The queue is ordered by generation time descending. Each queue item shows the output type badge, the scope entity, the generated time, and a review button.
The notification bell in the top bar shows a badge count of unacknowledged advisory outputs. Clicking the bell opens the advisory queue directly.
7. Branch and Compile Workflow
The branch and compile workflow surfaces are available to Architect and Reviewer roles. They provide visibility into the branch lifecycle — from creation through compile, review, and merge — without requiring architects to leave the Workbench for GitHub.
7.1 Branch Switcher
The branch selector in the top bar shows the active branch. Clicking opens a dropdown with:
Main branch entry with last compile time and status
All active branches grouped by intent (project, proposal, experiment, migration)
Per-branch: branch name, intent badge, compile status dot, review status badge (draft / submitted / approved / rejected), last activity time
Create new branch button (Architect only)
7.2 Create Branch
The Create Branch form is accessible from the branch switcher dropdown. It creates the branch in GitHub via the GitHub API and records the branch intent in the EAMS branch registry.
7.3 Compile Status Panel
The compile status panel is accessible from the status bar compile indicator and from the branch detail view. It shows the current compile run state, or the last compile outcome if no compile is in progress.
7.4 Branch Diff View
The branch diff view is accessible from the Branches section in the left panel. It shows the structural diff between the active branch and main: entities added, modified, and removed, with field-level change detail for modified entities.
7.5 Review Submission
The review submission flow is accessible from the branch detail view. It packages the compile report and diff into a review record and notifies the designated reviewers.
Architect clicks Submit for Review in the branch detail view.
Pre-submission checklist is evaluated: (a) last compile must be success or partial success — failed compile blocks submission; (b) no unresolved conflicts — unresolved conflicts produce a warning, not a block; (c) all advisory outputs of type compliance assessment and risk identification must be acknowledged — unacknowledged outputs block submission.
If checklist passes, the review submission form opens: reviewer selection (multi-select from architects list), review notes (optional), and a link to the branch diff view.
Architect submits. A review record is created in DynamoDB. Reviewers receive an email notification with a direct link to the branch diff view in the Workbench.
Reviewers access the branch diff via the Workbench. They can leave comments (stored in the review record), approve, or request changes.
Architect merges the branch via GitHub once the review is approved. Post-merge main compile runs automatically via GitHub Actions.
8. Search
Global search is accessible via the search input in the top bar (Cmd/Ctrl+K) and the Search section in the left panel. It queries the Architecture API's /search endpoint, which runs against the OpenSearch eams-search index.

10.2 Accessibility
Keyboard navigation: All interactive elements (navigation tree, diagram nodes in SVG, form inputs, advisory queue items) are keyboard-navigable. Focus ring visible on all focusable elements. Diagram supports keyboard zoom and pan.
Screen reader support: Entity detail panels, form fields, and advisory outputs have appropriate ARIA labels. Diagram SVG nodes have aria-label attributes with entity name and kind. Dynamic content updates (stale indicators, compile status changes) announce via aria-live regions.
Colour contrast: All text meets WCAG AA contrast ratios against their backgrounds. Status badges and compliance indicators use both colour and text labels — not colour alone.
Responsive layout: The split-panel layout supports viewport widths from 1024px upward. Below 1024px the left panel collapses to an icon rail. The Workbench is not designed for mobile viewports.

Lazy loading and performance
The tree does not load all entities on initial render. Expanding a domain node fetches entity kind counts (fast — one OpenSearch aggregation). Expanding a kind node fetches the first 50 entities in that kind for that domain. Scrolling to the bottom of an expanded kind node loads the next 50.
The global entity count is shown in the top bar: '847 entities across 5 domains'. This is updated on every compile completion event.

Rendering Technology | Role | Notes
D3.js (force / hierarchy layouts) | In-Workbench interactive rendering | Renders from canonical render graph JSON. Handles all layout algorithms specified in the viewpoint definition. Node and edge styling from render graph display properties.
Structurizr DSL | Export only | Projected by the Viewpoint Engine's Structurizr projector. Downloaded as a .dsl file or copied to clipboard.
Mermaid | Export + Confluence embed | Projected by the Mermaid projector. Used for Confluence page embedding via the Mermaid macro.
D2 | Export only | Projected by the D2 projector. Downloaded as a .d2 file.
SVG / PNG | Export only | SVG captured from the rendered D3 diagram. PNG is SVG rasterised at 2x resolution.
Why not embed Structurizr or Mermaid for rendering?
Embedding a Structurizr or Mermaid renderer as an iframe would make every diagram interaction (node click, hover, tooltip) require cross-frame communication, which is fragile and slow.
The canonical render graph JSON contains everything needed to render the diagram client-side: node positions (via layout hints), display properties (colours, shapes, badges), and relationship styling. D3.js rendering over this JSON gives the Workbench full interactivity without renderer coupling.
The export formats (Structurizr DSL, Mermaid, D2) are generated by the Viewpoint Engine on demand and are always consistent with the rendered view.

Form commits directly to the active branch
The edit form commits to whatever branch is currently active in the Workbench. If the architect is on main, the commit goes to main. If on a project branch, the commit goes to the project branch.
The form shows a prominent branch indicator ('Committing to: main') to prevent accidental commits to the wrong branch. Committing to main from the Workbench requires an additional confirmation step with a warning.
The form does not support committing to a new branch (branch creation is a separate action). Architects who want to create a branch from their edit should use the branch workflow before editing.

VS Code extension scope
The VS Code EAMS extension provides: YAML schema validation against the EAMS DSL schema, eams_id autocompletion (searches the compiled entity registry), rel_type autocompletion with valid values per kind pair, inline compliance hints, and the eams validate on-save trigger.
The VS Code extension is specified separately and is outside the scope of this document. Its presence is assumed for architects doing complex edits.

Advisory outputs do not expire
Advisory outputs remain in the queue indefinitely until acknowledged. They do not expire or auto-acknowledge.
If the entity an advisory output was generated for is subsequently modified (a new compile changes its records), the advisory output is marked stale with a 'Generated before last compile — entity may have changed' indicator. The architect can re-generate or acknowledge the stale output with a noted disposition.

Search Feature | Behaviour
Type-ahead | Results appear as the architect types, after a 200ms debounce. Minimum 2 characters.
Full-text | Matches entity name, description, and key metadata fields. Ranked by relevance score from OpenSearch BM25.
Faceted filtering | Left panel Search section shows facet counts: domain, kind, status, criticality. Clicking a facet adds it as a filter. Multiple facets are ANDed.
Scope limitation | Search results are scoped to the active branch. Main branch search always available; branch-specific search shows branch entities merged with main.

Concern | Technology | Rationale
Framework | React 19 | Component model well-suited to the panel layout and entity detail sections. Large ecosystem for diagram and data grid components.
State management | Zustand | Lightweight global state for active branch, user session, advisory queue counts. Avoids Redux boilerplate for a relatively simple state shape.
Data fetching | React Query (TanStack Query) | Handles caching, background refresh, and staleness for Architecture API calls. Pairs well with the entityUpdated subscription pattern.
Diagram rendering | D3.js v7 | Force layout for integration maps. Hierarchical layout for C4 diagrams. Custom SVG rendering over canonical render graph JSON.
Real-time updates | GraphQL subscription via AppSync WebSocket | entityUpdated subscription drives stale indicators and compile status updates.
Styling | Tailwind CSS + custom design tokens | Consistent spacing, colour, and typography without a heavy component library. EAMS design tokens map to the document palette.
Authentication | AWS Amplify (Cognito) | SAML/OIDC federation. Role-based access via Cognito groups. Token refresh handled by Amplify.
GitHub API integration | Octokit (REST) | Used for edit form commits and branch creation. Authenticated via OAuth app with per-user token.
