1. Introduction
The EAMS DSL (Enterprise Architecture Management System Domain-Specific Language) is the authoritative source of truth for all architecture data in the EAMS ecosystem. It is a YAML-based, Git-versioned language designed to be authored and maintained by enterprise architects, compiled by an automated pipeline, and consumed by downstream systems including the Ardoq graph database, the Architecture API, and the Visualization layer.
The core proposition: Architects write structured data files. Diagrams, wiki pages, and advisory outputs are derived from that data. The DSL file is the architecture — not the diagram.
1.1 Design Decisions
One file per entity: Each entity (system, container, process, capability, etc.) lives in exactly one file. File path encodes entity type and domain. Git diffs are meaningful at the entity level.
Relationships in domain files: Relationships between entities are declared in separate relationship files, one per domain. This eliminates merge conflicts between architects working on different systems in the same domain.
Human-readable IDs: Entity IDs are architect-assigned slugs (e.g., system.claims-processing), not database-generated UUIDs. IDs must be stable — once assigned, they do not change.
Rich relationship semantics: Relationship types (uses, exposes, stores, governs, etc.) are explicit in the DSL. They compile to Ardoq reference types with semantic metadata as attributes, keeping Ardoq viewpoint definitions manageable.
Layered completeness: A minimal valid entity requires only five fields. All other attributes are optional and additive. An incomplete entity is better than no entity.
Branching carries intent: Git branches are annotated with EAMS branch metadata declaring whether a branch represents a proposal, a project change, or an experiment. This metadata drives Workbench behavior and the approval workflow.

3.2 System (C4 Level 1)
A System represents a software system at the highest level of C4 abstraction — a black box delivering functionality to users or other systems. Systems map directly to application records in Salesforce TPM.
3.3 Container (C4 Level 2)
A Container is a deployable unit within a System — a web application, API, database, message broker, background worker, etc. Containers are where most architectural detail lives: technology choices, NFR profiles, and data storage.
3.4 Component (C4 Level 3)
Components are logical building blocks within a Container. They are optional — not every Container needs Components modeled. Components are appropriate when the internal structure of a Container is architecturally significant: a complex rules engine with distinct processing stages, a data ingestion container with multiple transformation steps, etc.
3.5 Infrastructure Node
Infrastructure Nodes represent physical or virtual infrastructure resources: Kubernetes clusters, managed databases, message brokers, CDN distributions, etc. Nodes are linked to containers via the container's infrastructure attribute and via deployment relationships in the relationships file.
3.6 Business Capability
Capabilities represent what the business does, expressed independently of how or which system implements it. Capabilities are deliberately lightweight — architects do not need business architecture training to capture a useful capability record.
3.7 Business Process
Process entities link BPMN 2.0 process models to the architecture graph. The BPMN file is the authoritative process definition — the DSL entity is the metadata envelope that connects the process to systems, capabilities, rules, and actors.
3.8 Data Entity
Data Entities are business-level data concepts — not physical tables. They represent the key information objects that the enterprise creates, maintains, and uses. Data Entities link upward to Business Capabilities and Processes, and downward to Data Products and physical models in Erwin.
3.9 Architecture Decision
Architecture Decisions capture significant design choices in ADR format. They are first-class entities in the metamodel — linked to the components they constrain, queryable by status, and surfaced in the Workbench alongside the entities they govern.
4. Relationships File Schema
Relationships between entities live in domain-scoped relationship files. Each file covers all relationships where at least one participant belongs to the named domain. A relationship between a Claims system and a Policy system lives in claims.relationships.yaml by convention — the domain of the source entity determines the file.

5.3 Branch Diff Semantics
The EAMS compiler produces a structured diff report when compiling a non-main branch. The diff report is the primary input to architecture review. It answers: what has changed, what has been added, what has been removed, and what standards violations have been introduced?
Entity additions: new entity files present in branch but not in base
Entity modifications: entity files present in both, with changed fields
Entity removals: entity files removed (or status changed to deprecated/retired)
Relationship additions: new relationships in domain relationship files
Relationship removals: relationships present in base but absent in branch
Standards violations: new entities or relationships that fail compliance checks
NFR regressions: containers whose NFR profile is weaker than the base branch

8.2 Workflow: Adding a New System to a Project Branch
Create a project branch from main: git checkout -b project/claims-rt-adjudication
Add .eams/branch.yaml with intent: project and project tracking information
Create the system entity file in systems/<domain>/<slug>.system.yaml
Create container entity files in containers/<domain>/<system-slug>/<slug>.container.yaml
Add relationships to relationships/<domain>.relationships.yaml
Run eams validate locally to check for errors before pushing
Push branch — CI pipeline runs schema validation, referential integrity, and compliance checks
Review the diff report generated by the compiler in the pull request
Submit for architecture review — update .eams/branch.yaml review.status to in-review
After review approval, update review.status to approved and merge to main

Scope of this document
This specification defines: DSL syntax, file organization, entity schemas, relationship schemas, branch metadata, and the compiler contract.
It does not define: the Ardoq mapping layer (covered in the Integration Specification), the viewpoint query language (covered in the Viewpoint Engine Specification), or the CI/CD pipeline implementation (covered in the Platform Engineering Guide).
System | Relationship to DSL
Ardoq | Compilation target. The DSL compiler writes entities and relationships to Ardoq via its REST API. Ardoq is never written to directly by architects.
Architecture API | The API reads from Ardoq (the compiled graph). The DSL is upstream of the API — changes flow DSL → Ardoq → API.
Git Repository | The DSL repository is the durable source of truth. Ardoq is a queryable index of the DSL content, not an independent record.
Architect Workbench | The Workbench can read DSL files directly from Git for editing, and displays diff views between branches. It does not write DSL directly — it generates DSL files for architect review and commit.
Confluence | Artifact generation reads from the compiled graph via the Architecture API. Confluence content is derived, not authoritative.
Salesforce TPM | TPM application records are referenced from DSL system entities via a tpm_id attribute. The sync integration is read-from-TPM, enrich-in-DSL — TPM metadata is not overwritten.

File naming convention
Format: <slug>.<entity-type>.yaml
Slug: lowercase, hyphen-separated, globally unique within entity type.
Entity type suffix is mandatory — it enables tooling to identify entity type without parsing the file.
Examples: claims-processing.system.yaml  |  adjudication-api.container.yaml  |  ADR-0042-rules-engine-selection.decision.yaml
Entity Type | Prefix | Example ID
System (C4 L1) | system. | system.claims-processing
Container (C4 L2) | container. | container.adjudication-api
Component (C4 L3) | component. | component.claim-validator
Infrastructure | infra. | infra.claims-eks-cluster
Business Capability | capability. | capability.claim-adjudication
Business Process | process. | process.first-notice-of-loss
Data Entity | entity. | entity.claim-record
Data Product | product. | product.claims-data-product
Data Contract | contract. | contract.claims-feed
Actor | actor. | actor.claims-adjudicator
Business Goal | goal. | goal.reduce-claims-cycle-time
Architecture Decision | decision. | decision.ADR-0042
ID stability rule
Once an entity ID is committed to the main branch, it MUST NOT change.
Ardoq component IDs, API URLs, viewpoint definitions, and cross-references all depend on stable IDs.
If an entity needs renaming, use the display name field. The ID remains frozen.
If an entity is retired, set status: retired — do not delete the file from Git history.

C4 Level 3 usage guidance
Model Components when: the container has distinct internal subsystems with separate teams, patterns, or data concerns.
Do not model Components when: the container is a simple CRUD API or single-responsibility service. Premature L3 modeling adds noise without insight.
Components do not appear in C4 Context or Container diagrams — they are only visible in Component-level viewpoints.

You do not need a formal business architecture process to create a useful capability record.

description: Selection of Drools as the embedded rules engine for the Claims Adjudication container.

adr:
context: Claims adjudication requires evaluation of hundreds of policy rules that change quarterly. A dedicated rules engine separates rule authoring from application code deployment.
decision: Adopt Drools (Apache-licensed) as an embedded rules engine within the Adjudication API container.
alternatives_considered:
- name: Easy Rules  rationale: Insufficient expressiveness for complex coverage scenarios
- name: AWS Lambda rules  rationale: Latency and operational complexity unacceptable for synchronous adjudication
consequences: Rules are authored in DRL. Team requires Drools training. Rule deployment decoupled from API deployment.
review_trigger: If adjudication rule volume exceeds 1000 active rules or latency SLA cannot be met.

affects:
- container: container.adjudication-api
- component: component.claim-validator
- standard: standard.approved-jvm-libraries

Why separate relationship files?
Eliminates merge conflicts: two architects can update different systems in the same domain simultaneously without touching each other's entity files.
Makes the integration map explicit: claims.relationships.yaml is the definitive map of how the Claims domain connects to everything else.
Enables domain-scoped relationship review: architecture review can assess a domain's integration surface in a single file.

exposes | A → B | container → interface (self) | interface | Declares interface provided by container
publishes-to | A → B | container/system → container/system | integration | Async event/message publication
subscribes-to | A → B | container → container/system | integration | Async event/message subscription
stores-in | A → B | container → container (db/store) | data-flow | Write access to a data store
reads-from | A → B | container → container (db/store) | data-flow | Read-only access to a data store
deployed-on | A → B | container → infra | deployment | Container running on infrastructure node
supported-by | A → B | capability → system | realization | Capability technically enabled by system
realized-by | A → B | goal → capability | realization | Goal achieved through capability
invokes | A → B | process → system/container | invocation | Process step calls a system
governed-by | A → B | process/component → rule | governance | Entity subject to a business rule
constrains | A → B | decision/standard → any | governance | ADR or standard constrains entity
produces | A → B | system/container → data product | data-flow | System publishes a data product
contracts-with | A ↔ B | data product → contract | contract | Bidirectional contract linkage

Branch rules enforced by compiler
A .eams/branch.yaml file is required on all non-main branches. Compiler fails without it.
intent: project branches must include a project.id referencing a valid project record.
intent: proposal branches must list at least one reviewer in review.reviewers.
Merge to main is blocked unless review.status = approved (for proposal and project branches).
The base_branch field is immutable after first commit — do not rebase and change it.
Stage | Description | Failure Behavior
1. Schema validation | Every entity file and relationship file is validated against the metamodel JSON Schema. Required fields checked. Enum values checked. ID format checked. | Hard failure — commit rejected. Error report lists all violations with file and line number.
2. Referential integrity | All entity references (parent_system, capabilities, goal refs, relationship from/to) are resolved against the full entity graph. | Hard failure — dangling references are not permitted. Error lists unresolved IDs.
3. ID stability check | Checks that no entity ID present in the base branch has been changed. | Hard failure on main branch. Warning on feature branches.
4. Standards compliance | Each entity is checked against applicable standards from the standards catalog. | Soft failure — produces findings report. Does not block compile. Exceptions require ADR documentation.
5. NFR regression check | Containers compared to base branch NFR profiles for regressions in availability, RTO, RPO, or data classification downgrade. | Soft failure — produces regression report. Requires architect acknowledgment to proceed.
6. Ardoq write | Validated entities and relationships are written to Ardoq via API. Entities are upserted by qualified ID. Relationships are upserted by relationship ID. | Hard failure if Ardoq API is unavailable or rejects writes. Retried 3x with backoff before failing.
7. Diff report | For non-main branches, a structured diff report is generated and stored as a build artifact. Posted to the pull request as a comment summary. | Non-blocking — diff report failure does not fail the build.
8. Search index update | OpenSearch index is updated with all changed entities and relationships. | Non-blocking — search lag is acceptable. Indexed asynchronously.
DSL Construct | Ardoq Target | Mapping Notes
Entity file | Ardoq Component | Component type maps to entity kind. Qualified ID stored as a component attribute for stable cross-reference.
Entity domain | Ardoq Workspace | One Ardoq workspace per domain. Cross-domain relationships use Ardoq cross-workspace references.
Relationship (any type) | Ardoq Reference | All relationship types map to one of five Ardoq reference types: integration, data-flow, deployment, realization, governance. The DSL relationship type is stored as a reference attribute.
Interface declaration | Ardoq Component | Interfaces declared on containers are compiled as child components of type Interface. Interface ID is the container ID + fragment.
NFR profile | Component attributes | All NFR fields are stored as typed attributes on the container component. Enables Ardoq viewpoint filtering by NFR values.
Branch (non-main) | Ardoq Scenario | Non-main branches compile to a named Ardoq Scenario for visualization. Scenario name includes branch intent and branch name.
Standards compliance finding | Component attribute | Compliance status and finding list stored as attributes. Enables viewpoints that highlight non-compliant components.
The Ardoq Viewpoint complexity solution
Ardoq Viewpoints become complex when relationship semantics are encoded as distinct Ardoq reference types.
The EAMS approach: all integration relationships share one Ardoq reference type ('integration'). The semantic type (uses, consumes, publishes-to, etc.) is a reference attribute.
Viewpoints filter on the semantic attribute rather than requiring a separate reference type per semantic. This keeps Viewpoint definitions manageable while preserving full semantic richness in the DSL.
