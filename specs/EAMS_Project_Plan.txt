1. Executive Summary
This document defines the strategic architecture and project plan for the Enterprise Architecture Management System (EAMS) — a transformational initiative to replace fragmented, manually-maintained architecture artifacts with a unified, data-driven ecosystem. The initiative addresses long-standing limitations across the current toolchain (Ardoq, Salesforce TPM, Confluence, Gliffy) and establishes a modern platform grounded in four foundational principles:
Architecture information treated as first-class data — versionable, queryable, and governed
A pivot from 'building diagrams' to 'developing data', with diagrams as derived outputs
Deep traceability from business goals and user experience through technical implementation
LLM-augmented advisory capabilities embedded into the architecture practice workflow
The system will be anchored on Ardoq's graph database as the authoritative backend, extended with a Git-backed DSL layer for versioning, a purpose-built API and viewpoint engine, and an Architect Workbench portal. The initiative targets a functional architecture in place by Q4 2026, with ongoing content migration continuing into subsequent years.

Existing Architecture Data: Thousands of existing Ardoq components and relationships are migrated and enriched under the new metamodel. Existing Confluence and Gliffy artifacts are catalogued, assessed, and progressively migrated.

3.2 Core Architectural Decisions
ADR-001: Ardoq as Graph Backend, Abstracted by API Layer
Ardoq's graph database is the most powerful asset in the current landscape. Rather than replacing it, we build an abstraction layer — a purpose-built API and ingestion pipeline — that insulates the rest of the ecosystem from Ardoq's operational complexity. Direct Ardoq UI usage is reduced to administrative and diagnostic scenarios.
ADR-002: Git-backed DSL as Source of Truth for Architecture Data
Architecture entities, relationships, and attributes are authored in a structured DSL committed to a Git repository. This provides the versioning, branching, and merging semantics that Ardoq's Scenario editor cannot adequately support. The Git repository compiles into the Ardoq graph — Git is the durable record, Ardoq is the queryable index.
ADR-003: C4 Model as Primary Technical Architecture Language
C4 (Context, Containers, Components, Code) is adopted as the primary modeling language for technical architecture. C4 maps naturally to Ardoq's component/reference model and provides stakeholder-appropriate views at each level. ArchiMate constructs are selectively incorporated for cross-domain concepts (motivation, strategy, physical) where C4 has no equivalent.
ADR-004: BPMN as Business Process Modeling Language
BPMN 2.0 is adopted for business process modeling. BPMN processes are linked to business capabilities, business rules, and C4 system boundaries to enable end-to-end traceability. BPMN artifacts are stored in the Git-backed DSL and linked to Ardoq entities.
ADR-005: Viewpoints as Queries, Not Diagrams
Following the Views and Beyond framework, a viewpoint is defined as a query over the architecture graph combined with a rendering specification. Diagrams are derived outputs — they are never the authoritative record. This enables unlimited stakeholder-specific visualization without divergence from the authoritative data.
ADR-006: AWS-native Infrastructure
The EAMS platform is built on AWS, aligned to the company's existing cloud posture. Key services: S3/CodeCommit for DSL versioning, Lambda/ECS for processing pipelines, API Gateway for the architecture API, EventBridge for signal ingestion, Bedrock for LLM advisory capabilities, and OpenSearch for full-text architecture search.

4. Information Metamodel
4.1 Metamodel Purpose
The information metamodel is the schema of schemas for EAMS. It defines what entity types exist, what relationships are valid between them, what attributes each entity type carries, and what constraints govern data quality. All architecture tooling is downstream of the metamodel.

4.3 Key Relationship Types
Relationships in the EAMS metamodel are first-class entities with their own attributes — not just graph edges. This resolves the Ardoq Viewpoint complexity problem by encoding relationship semantics in the metamodel rather than in viewpoint definitions.

5. Project Plan
5.1 Approach
The EAMS initiative is organized into six phases. Phases 0 and 1 are strictly sequential prerequisites. Phases 2 through 5 are designed for parallel execution with defined integration checkpoints. The cross-cutting workstream runs continuously throughout.
5.2 Phase Overview
5.3 Phase Detail
Phase 0 — Foundation & Discovery  (Weeks 1–6)
0.1 Current State Audit: Catalog all architecture artifacts across Confluence, Gliffy, Ardoq, and Salesforce. Document volume, quality, and coverage gaps. Interview architects across application, infrastructure, security, and any embedded business architecture functions.
0.2 Pain Point Analysis: Map specific workflow failures — not platform-level complaints — to determine root causes and prioritization. Identify which current Ardoq workspaces, component types, and references are salvageable vs. require restructuring.
0.3 Metamodel Scoping Workshop: Facilitated session with practicing architects to agree on v1 entity types, relationship types, and attribute sets. This is the single highest-value activity in Phase 0.
0.4 Technology Candidate Evaluation: Evaluate Git hosting (CodeCommit vs. GitHub Enterprise), DSL format (YAML/HCL/custom), diagram rendering candidates (Structurizr, D2, Mermaid, custom), API framework, and LLM service (Bedrock preferred).
0.5 ADR Initialization: Establish the ADR registry. Document Architectural Decision Records for all choices made in Phase 0.
Phase 0 Exit Criteria: Signed-off metamodel scope document. Technology candidate decision matrix completed. Phase 1 scope baselined.
Phase 1 — Core Metamodel & Data Architecture  (Weeks 6–18)
1.1 Metamodel v1 Implementation: Define and implement the full v1 metamodel. Produce the formal metamodel schema — entity types, relationship types, attribute definitions, cardinality constraints, and validation rules.
1.2 Git-backed DSL Design: Design the DSL syntax for authoring architecture entities and relationships. Tooling: schema validation on commit (pre-commit hooks), CI pipeline that compiles DSL to Ardoq graph via API.
1.3 Ardoq Schema Alignment: Map the EAMS metamodel to Ardoq workspaces, component types, and reference types. Enforce the metamodel through the Ardoq API rather than relying on Ardoq's native UI conventions.
1.4 Versioning & Branching Model: Implement Git branching strategy for architecture evolution. Define merge semantics for competing architecture branches. Build tooling to diff architecture states between branches.
1.5 Data Migration Pipeline v1: Build importers for existing Ardoq components. Assess and categorize existing content against the new metamodel. Execute initial migration of highest-value, highest-quality existing data.
Phase 1 Exit Criteria: Metamodel v1 schema published. DSL toolchain operational. Initial Ardoq migration complete for at least one architecture domain. Branching model demonstrated.
Phase 2 — API, Integration & Signal Layer  (Weeks 12–24)
2.1 Architecture API: REST + GraphQL API over the architecture graph. Read operations: query by entity type, relationship, domain, viewpoint. Write operations: validated against metamodel constraints. API-first design — all internal tooling consumes the same API.
2.2 Salesforce TPM Integration: Extend the existing Ardoq ↔ Salesforce integration to operate through the EAMS API layer. Bi-directional sync of application and infrastructure metadata. TPM remains the system of record for portfolio management; EAMS is the system of record for architecture relationships.
2.3 Signal Ingestion Pipeline: AWS EventBridge-based event bus for operational signals. Initial signal types: production incidents (ServiceNow/Jira), application performance metrics, technology lifecycle events (retirement notices, version changes). Signals are enriched with architecture context and surfaced to architects as evaluation candidates.
2.4 Artifact Generation Engine: Template-based generation of Confluence pages from architecture data. Initial templates: System Overview, Integration Specification, Data Flow. Generation triggered by DSL commit pipeline or on-demand via API.
2.5 Architecture Search: AWS OpenSearch index over all architecture entities, relationships, and attributes. Replaces ad-hoc Confluence search for architecture discovery. Faceted search by domain, lifecycle, owner, standard compliance.
Phase 3 — Viewpoint, Visualization & Portal  (Weeks 20–32)
3.1 Viewpoint Engine: Implement viewpoints as first-class metamodel objects: a named query (graph traversal expression) + a rendering specification (layout rules, element styling, relationship filtering). Any architect can define a new viewpoint without engineering assistance.
3.2 Diagram-as-Code Renderer: Evaluate and select the rendering target. Primary candidates: Structurizr DSL (strong C4 alignment, programmatic layout), D2 (modern, clean output, flexible). Implement a rendering pipeline: architecture graph query → intermediate representation → diagram output. Target channels: Confluence embed, SVG export, PDF, IDE plugin.
3.3 Architect Workbench Portal: Web application providing the primary architect interface. Features: entity browser (graph navigation), viewpoint browser, data entry and editing forms (validated against metamodel), DSL editor with preview, diff viewer for branch comparison, signal queue for architect review.
3.4 Multi-stakeholder View Library: Implement initial library of stakeholder-specific viewpoints: Executive (capability heat map), Business Analyst (process-to-system), Developer (container/component detail), Operations (infrastructure topology), Security (trust boundary and data flow).
Phase 4 — Business Architecture & SME Tooling  (Weeks 26–40)
This phase acknowledges that there is no centralized business architecture function. The tools must enable technical architects to capture MVP business context without requiring business architecture expertise. LLM assistance is load-bearing here.
4.1 Business Layer Metamodel Extension: Extend the v1 metamodel with the business layer entity types: Business Goal, Capability, Process (BPMN-linked), Rule, Term, Actor. Define traceability relationships from each business entity to the technical layer.
4.2 BPMN Integration: Select and integrate a BPMN authoring tool (Camunda Modeler preferred — open source, BPMN 2.0 compliant). BPMN files stored in Git alongside DSL. Parser extracts process elements and links them to metamodel entities.
4.3 LLM-guided Business Intake: Chat interface (powered by Amazon Bedrock) that guides technical architects through capturing business context. Structured prompts elicit: business capability description, owning business unit, key processes, important business rules, relevant KPIs. Outputs are validated and committed to the DSL as business layer entities.
4.4 Figma / UX Artifact Linkage: Figma file references linked to business processes and C4 systems. Metadata extraction from Figma API: screen names, user flows, linked components. Traceability chain: Screen → Process Step → Business Rule → System → Container.
4.5 Goal-to-Outcome Lineage: Implement the traceability chain from strategic goals through capabilities, processes, and systems to measurable outcomes. Lineage viewpoint provides executives with a coherent view of how IT investment delivers business value.
Phase 5 — Advisory, Governance & Standards  (Weeks 34–52)
5.1 Standards Migration: Migrate all technical architecture standards from Salesforce into the EAMS standards metamodel. Standards are linked to the entity types they govern. Compliance checking is automated as part of the DSL commit pipeline.
5.2 Automated Compliance Validation: On every DSL commit, run compliance checks against applicable standards. Non-compliant entities surface in the Architect Workbench as validation findings. Exception workflow: architect documents rationale, review board approves, exception recorded in ADR.
5.3 LLM-augmented Architecture Review: Bedrock-powered advisory layer that reviews submitted architecture data against standards, patterns, and best practices. Produces: compliance findings, pattern match suggestions, risk flags, and draft advisory commentary for architect review. Architects review, annotate, and approve before distribution.
5.4 Pattern Library: Catalog of recognized architecture patterns stored in the metamodel. Patterns are linked to the entity types they apply to. LLM advisory layer can suggest applicable patterns during architecture development.
5.5 Architecture Review Board Support: Workflow for architecture submission, review, approval, and publication. Review board members interact with architecture data directly in the Workbench rather than reviewing static diagrams. Full audit trail of review decisions.

Risk | Mitigation
Adoption failure — architects continue using Confluence/Gliffy manually | Invest heavily in Workbench UX. Make the data-driven workflow strictly easier than the manual workflow. Enforce data-first through governance processes, not just tooling.
Metamodel scope creep — every stakeholder wants their concept modeled in v1 | Strict v1 scope gate after Phase 0 workshops. Metamodel extensibility is built-in; defer non-core entities to v2 with a documented path.
Ardoq API limitations surface during integration | Phase 0 technology evaluation includes a detailed Ardoq API capability assessment. Design the abstraction layer to accommodate known limitations (rate limits, reference complexity).
LLM advisory output quality is insufficient for production use | LLM advisory is always human-reviewed before action. Phase 5 begins with a controlled pilot. Quality gates are defined before broad rollout.
Confluence migration creates parallel maintenance burden | New content goes into EAMS-generated pages from day one. Historical migration is batched and architect-reviewed. A clear cut-over date for each domain is set.
Engineering resource availability for initial build | Phases 0–1 are architect-executable. Engineering engagement is scoped to Phase 2+ and can be phased to match resource availability.
Data quality of existing Ardoq content is worse than expected | Phase 0 audit produces a data quality baseline. Migration is gated on quality thresholds. Low-quality content is quarantined rather than migrated blindly.
