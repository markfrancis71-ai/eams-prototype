1. Overview
The EAMS LLM Advisory Layer is the intelligence surface of the EAMS ecosystem. It synthesises the compiled architecture graph — available through the Architecture API context bundle endpoint — with large language model reasoning to produce six categories of advisory output: compliance assessment, risk identification, design alternatives, impact analysis, natural language explanation, and standards guidance.
The advisory layer is differentiated from a generic LLM chat interface in three critical ways. First, it operates on grounded context — every advisory output is anchored to actual entity data, relationship data, and standards definitions from the architecture graph, not trained model memory. Second, it enforces a human review gate calibrated to output risk — informational outputs surface directly, while assessments and recommendations require explicit architect acknowledgement before influencing any downstream artifact. Third, its prompt templates are versioned code — stored in the EAMS repository alongside the schemas and standards they reference, reviewed and tested like any other spec artifact.
1.1 Design Decisions
All six output types in v1: Compliance assessment, risk identification, design alternatives, impact analysis, natural language explanation, and standards guidance are all in scope. They are differentiated by risk tier, which drives context assembly depth, model routing, and review gate requirements.
Moderate review gate: Informational output types (explanation, standards guidance) surface directly in the Workbench without an acknowledgement step. Assessment output types (compliance, risk, impact, alternatives) require explicit architect acknowledgement before they can influence a downstream artifact such as a Confluence page, a review decision, or a branch comment.
Prompt templates as versioned repository artifacts: Prompt templates live in .eams/prompts/ in the EAMS repository alongside schemas, standards, and viewpoints. Changes go through the same Git review process. Templates are loaded at Lambda cold start and cached — no redeploy required for prompt updates, but changes are auditable and reversible.
Context bundle as the primary data interface: The advisory layer calls GET /entities/{id}/context-bundle to assemble entity context, rather than making direct OpenSearch or Ardoq calls. This keeps the advisory layer's data access decoupled from the storage layer and ensures it benefits from API caching.
Model routing by output type: Informational output types use a smaller, faster, cheaper model. Assessment output types use a larger model with stronger reasoning capability. This is configurable via the prompt template metadata — not hardcoded.
Structured output schemas: All six output types produce JSON-structured responses conforming to a defined schema, not free-text. Structure is enforced via JSON mode and schema validation. Free-text narrative is a field within the structured response, not the entire response.

2. Output Type Specifications
Each advisory output type has a defined risk tier, context assembly strategy, model routing, review gate requirement, and structured output schema. The six types are specified below.
2.1 Compliance Assessment
Evaluates whether an entity or a set of entities satisfies its applicable standards. The compiler's rule-based compliance check is authoritative for deterministic rules. The advisory layer adds value by identifying pattern-level violations — issues that arise from the combination of attributes and relationships rather than any single field value — and by synthesising compliance status across a domain into an executive-facing assessment.
2.2 Risk Identification
Identifies architectural risks carried by a system or domain. Unlike compliance assessment — which checks against defined standards — risk identification reasons across multiple dimensions: resilience, security posture, technical debt, operational complexity, and strategic alignment. It surfaces risks that do not have a corresponding standard rule but represent genuine architectural concerns.
2.3 Design Alternatives
Proposes two or three alternative architectural approaches to a stated design problem. This is the highest-risk output type: alternatives require the most context, carry the highest potential for confident but wrong suggestions, and require the most careful architect review. Accordingly, they receive the deepest context assembly, the largest model, and the most explicit review gate language.
2.4 Impact Analysis
Analyses the downstream and upstream impact of a proposed change to an entity or a set of entities. Distinct from the blast radius viewpoint — which shows static graph dependencies — impact analysis reasons about the consequence of specific changes: what other entities would need to change, what SLA contracts would be affected, what compliance status would change, and which teams would need to coordinate.
2.5 Natural Language Explanation
Produces a natural language description of an entity, a system, or a domain for a specified stakeholder audience. This is the lowest-risk output type — it is purely informational and does not make claims about correctness, risk, or compliance. It surfaces directly in the Workbench without a review gate, though architects can optionally review before publishing to Confluence.
2.6 Standards Guidance
Explains a standard and its requirements as they apply to a specific entity or entity type. Surfaces the standard's rules in plain language, describes what the entity must do to comply, and links to the relevant field requirements in the DSL schema. Low-risk, informational — no review gate.
3. Context Assembly
Context assembly is the most consequential engineering problem in the advisory layer. The quality of an advisory output is directly bounded by the quality and completeness of the context provided to the model. Too little context produces vague, generic advice. Too much context overflows the context window, incurs unnecessary cost, and degrades reasoning quality by burying relevant signal in noise.

4. Prompt Architecture
Prompts are structured artifacts, not ad-hoc strings. Each output type has a versioned prompt template stored in .eams/prompts/. Templates are YAML files with a metadata header and the prompt text as a structured multi-section document. The Lambda function loads and caches templates at cold start — prompt updates take effect on the next Lambda instance initialisation, typically within minutes, without a redeploy.

4.3 Prompt Versioning and Review
Versioning: Prompt templates carry a version field. When a template is updated, the version is incremented. Advisory outputs record the prompt version used, enabling correlation between output quality changes and prompt changes.
Review requirements: Prompt template changes go through a pull request review by at least one platform engineer and one practising architect. Changes to assessment prompts (compliance, risk) require review by two architects given their higher impact on advisory quality.
Testing: A prompt change PR must include a test fixture: a reference entity context and the expected output (or the delta in expected output quality). The eams prompt test command runs the new template against the fixture and diffs the output against baseline.
Rollback: Because templates are in Git, prompt rollback is a Git revert. The Lambda cache evicts on the next cold start (typically within 15 minutes) or is force-evicted via the eams prompt reload CLI command.

7.3 Feedback-Driven Prompt Iteration
Quality metrics are reviewed weekly by the platform engineer.
A rejection rate ≥ 15% on any prompt template triggers a prompt review meeting with two practising architects.
Rejected outputs with disposition notes are reviewed as a corpus — common themes in rejection notes drive prompt changes.
Prompt changes are tested against a fixture set of reference entities before merge. The fixture set grows as new rejection patterns are identified.
Model upgrades (e.g. upgrading from claude-sonnet-4-5 to a newer model) are evaluated by running the full fixture set through both models and comparing outputs against the baseline.

9. Performance & Reliability
9.1 Latency Targets
Latency is dominated by model inference time. Context assembly adds 200–800ms depending on context budget. The Architecture API context bundle call adds 50–300ms (cached). Lambda cold start adds up to 2s on the first request after a period of inactivity — mitigated by provisioned concurrency for the advisory Lambda during business hours.
9.2 Reliability Design
Model API failure: If the model API call fails, the advisory Lambda retries twice with exponential backoff (2s, 4s). After three failures, the request is marked failed with MODEL_API_UNAVAILABLE. The Workbench shows a retry option.
Schema validation failure: If the model response does not conform to the expected output schema, the advisory Lambda retries once with a schema correction prompt appended. If the second attempt also fails validation, the request is marked failed with INVALID_MODEL_OUTPUT.
Context assembly failure: If the Architecture API context bundle call fails, the advisory request is failed immediately with CONTEXT_UNAVAILABLE. No retry — the context bundle API has its own retry and circuit breaker.
Partial context: If the context bundle returns fewer fields than expected (e.g. a sparse entity), the advisory layer proceeds with a reduced context and reports data gaps in the output. Partial context is not treated as a failure.

What the advisory layer is not
It is not a general architecture chatbot. Queries are structured and scoped to the architecture graph context. Free-form chat is out of scope for v1.
It is not an authoritative compliance engine. The compiler's rule-based compliance checks are authoritative. The advisory layer provides synthesis, explanation, and identification of issues the compiler's deterministic rules may not catch — pattern-level risks, cross-cutting concerns, missing relationships.
It does not write DSL files. Advisory outputs are text artifacts surfaced in the Workbench. The architect decides whether to act on them and authors any resulting DSL changes.

Token estimates are soft limits
The context budget is a target, not a hard cutoff. The context assembly pipeline tracks token consumption as it builds the context and truncates or summarises lower-priority sections when the budget is approached.
Token counts are estimated using a character-based heuristic (1 token ≈ 4 characters) during assembly, then verified against the model's tokenizer before dispatch. If the assembled context exceeds budget by more than 10%, the assembler re-runs one priority tier lower.
Priority | Content Type | Inclusion Rule | Approx Tokens
P1 | Primary entity — all fields | Always included in full | 300–600
P1 | Directly applicable standards — full text | Always included (up to 3 standards full text; remainder summarised) | 500–2,000
P1 | Active compliance findings on primary entity | Always included in full | 200–500
P2 | Direct relationships (1-hop) — full relationship records | Included in full up to 20 relationships; remainder summarised as counts by type | 600–2,000
P2 | Related entities (direct relationship targets) — key fields only | id, name, kind, domain, status, criticality. Full records omitted to save budget. | 500–1,500
P3 | Accepted decisions (ADRs) linked to primary entity | Included up to 3 full ADRs; remainder as title+status only | 300–1,200

Context Assembly — Pseudocode
# Input: entity_id, output_type, branch
# Output: assembled_context (string), token_count, data_gaps (list)

function assembleContext(entity_id, output_type, branch) {

# Step 1: Fetch context bundle from Architecture API
const bundle = await GET /entities/{entity_id}/context-bundle?depth=2&branch={branch}
const budget = CONTEXT_BUDGETS[output_type]

# Step 2: Assemble P1 context — always included
let ctx = formatEntity(bundle.entity)                  # ~400 tokens
ctx += formatStandards(bundle.applicable_standards, full=3)   # ~1,200 tokens
ctx += formatFindings(bundle.compliance_findings)             # ~300 tokens

# Step 3: Assemble P2 context — included if budget permits
if (estimateTokens(ctx) + 2600 < budget) {
ctx += formatRelationships(bundle.relationships, full=20)
ctx += formatRelatedEntities(bundle.relationships, keyFieldsOnly=true)
}

# Step 4: Assemble P3 context — decisions and signals
if (estimateTokens(ctx) + 1600 < budget) {
ctx += formatDecisions(bundle.decisions, full=3)
ctx += formatSignals(bundle.signals, last=5, summaryOnly=true)
}

# Step 5: Assemble P4 context — 2-hop summary
if (estimateTokens(ctx) + 600 < budget) {
ctx += formatTwoHopSummary(bundle.relationships)
}

# Step 6: Identify data gaps — fields absent from DSL that affect output quality
const dataGaps = identifyDataGaps(bundle, output_type, REQUIRED_FIELDS[output_type])

return {
context: ctx,
tokenCount: estimateTokens(ctx),
dataGaps: dataGaps
}
}

- The entity's DSL definition, including all declared fields and relationships
- The full text of all applicable standards
- Active compliance findings already identified by the compiler's rule engine
- Related entity summaries providing dependency context

Assessment rules:
1. Base every finding on evidence from the provided entity data. Do not invent fields, relationships, or attribute values.
2. Distinguish deterministic findings (the compiler also catches this) from pattern findings (cross-cutting concerns the compiler's rule engine does not evaluate).
3. Report data gaps where absent DSL fields would materially affect the assessment.
4. Self-assess your confidence as high / medium / low. Low = significant data gaps or ambiguous evidence.
5. Produce a remediation priority list ordered by severity and ease of remediation.
6. Respond ONLY with valid JSON conforming to the compliance_assessment_v1 schema. No preamble. No markdown.

Temperature guidance
Assessment outputs (compliance, risk, impact) use low temperature (0–0.2) to maximise determinism and evidence-anchoring. The same entity assessed twice should produce the same findings.
Generative outputs (design alternatives) use moderate temperature (0.4) to produce genuinely distinct alternatives rather than variations on the same approach.
Explanations use moderate temperature (0.3) to produce natural-sounding prose rather than robotic field enumeration.

Model IDs are configuration, not code
The model_id field in each prompt template controls which model is called. Upgrading from claude-sonnet-4-5 to a newer model requires updating the prompt template file and merging via the normal review process — no Lambda code change, no redeploy.
Per-environment overrides: the EAMS_MODEL_OVERRIDE environment variable can force all requests to a specific model. Used in development to always route to a cheaper model and avoid cost runaway during testing.

Disposition | Meaning | Effect
accepted | Architect agrees with the assessment and intends to act on all findings. | Output can be published. Findings are added to the entity's advisory action queue in the Workbench.
accepted_with_modifications | Architect agrees with some findings but disputes or modifies others. Disposition note is required. | Output can be published with a note indicating modifications. Only accepted findings added to action queue.
rejected | Architect disputes the assessment. Rejection reason required. | Output is not published. Rejection recorded. If a finding recurs in the next compile, it is flagged to the platform engineer for prompt review.
noted | Architect has reviewed but does not intend to act immediately. Note required. | Output recorded. No action queue items created. Advisory resurfaces on next compile if findings persist.

Async model and polling
All advisory requests are async. The Workbench uses long-polling (GET /advisory/{id} with up to 30s wait) to retrieve completed outputs. The CLI uses synchronous polling with a progress indicator.
Typical completion times: explanation and standards guidance 1–3s; compliance and risk 4–10s; design alternatives 8–20s.
Failed requests include an error field with a structured error code. The most common failure is CONTEXT_TOO_SPARSE — the entity has too few fields populated to produce a useful advisory output. The advisory layer returns the data gap list even on failure.
