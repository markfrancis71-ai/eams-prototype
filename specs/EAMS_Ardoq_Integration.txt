1. Overview
The EAMS Ardoq Integration Layer is the boundary between the EAMS ecosystem and the Ardoq graph database. It owns all reads and writes to Ardoq. No other EAMS component — not the compiler pipeline, not the Workbench, not the Architecture API — interacts with Ardoq directly. This isolation is the foundational design decision that protects the rest of the ecosystem from Ardoq's operational constraints.
The integration layer is responsible for five concerns, each addressed by a distinct sub-component:

2.2 EAMS Entity → Ardoq Component Mapping
All EAMS entity types map to Ardoq Components. The entity type is encoded in the Ardoq component type, using the EAMS:: namespace prefix to prevent collision with any existing Ardoq component types in the tenant.
2.3 Component Hierarchy — C4 Nesting in Ardoq
Ardoq's component parent/child relationship is used to represent the C4 containment hierarchy. This enables Ardoq's native tree navigation and tree-style viewpoints to work correctly without custom configuration.

Fifteen DSL relationship types compress to five Ardoq reference types. The DSL relationship type is preserved as the eams_rel_type field on the Ardoq reference. This is the mechanism that keeps Ardoq Viewpoint definitions manageable — viewpoints filter on eams_rel_type rather than requiring a distinct Ardoq reference type per semantic.

3. ID Registry
The ID Registry is the most operationally critical component of the integration layer. It maintains the durable, bidirectional mapping between DSL qualified IDs and Ardoq internal _id values. Without this mapping, the upsert engine cannot reliably locate existing components to update — it would be forced to query Ardoq by name, which is neither reliable nor performant.

5. Upsert Engine
The Upsert Engine is the core of the integration layer. It receives a validated, compiled entity graph from the DSL compiler and writes it to Ardoq. It operates in two modes: diff-aware (default) and force-full-sync (explicit flag).
5.1 Diff-Aware Mode
In diff-aware mode, the engine only writes components and references that have changed since the last successful compile. Change detection uses a content hash of the compiled entity's Ardoq field values, stored in DynamoDB alongside the Ardoq ID.
The content hash is a SHA-256 of the JSON serialization of the entity's Ardoq field values (excluding eams_git_sha and eams_compiled_at which change on every compile regardless of content). This means a pure metadata change like a commit timestamp does not trigger a write to Ardoq.
5.2 Write Ordering — Dependency Resolution
Ardoq references require both the source and target components to exist before the reference can be created. The upsert engine processes entities in strict dependency order to avoid write failures on cross-component references.
5.3 Force-Full-Sync Mode
Force-full-sync bypasses the content hash check and writes every entity in the compiled graph. It is invoked with eams compile --force-full-sync. It should be used for:
Initial bootstrap: First compile of a new domain, new branch workspace, or fresh Ardoq environment.
Recovery: After a detected drift between the DSL state and Ardoq state (identified by the nightly audit job).
Schema migration: After changes to the Ardoq component type field schema (new fields added, field types changed) that require re-writing all component field values.
Explicit architect request: Via the eams CLI when an architect suspects inconsistency and wants a clean reconciliation.
5.4 Component Write Operations
CREATE — New component
UPDATE — Existing component
RETIRE — Status transition to retired
Retiring an entity in EAMS does not delete the Ardoq component. Deletion of architecture history is destructive and non-recoverable. Instead, the upsert engine:
Sets eams_status to retired on the component
Sets eams_compiled_at to the retirement compile timestamp
Removes all outbound references from the retired component (references from other components to the retired component are preserved as historical context, flagged with eams_target_retired: true)
Adds the component to a dedicated Ardoq tag: EAMS::Retired
5.5 Batched Write Strategy
The upsert engine never makes individual sequential API calls. All writes are batched and executed with controlled concurrency to respect Ardoq API rate limits.
5.6 Reference Write Operations
Cross-workspace references
When a relationship in the DSL crosses domain boundaries (e.g. Claims container consuming a Policy interface), the Ardoq reference spans two workspaces. Ardoq supports this via its cross-workspace reference API, which requires both the source workspace ID and the target workspace ID.

6.1 Scenario Management (Proposal & Experiment Branches)
Ardoq Scenarios are overlays on an existing workspace. They record additions, modifications, and deletions as a delta against the baseline workspace state. The EAMS compiler writes branch changes as Scenario entries rather than modifying the baseline.
6.2 Full Workspace Copy (Project Branches)
Project branches receive a full Ardoq workspace copy. This gives architects and reviewers the full Ardoq Viewpoint experience — running viewpoints, generating reports, validating NFR profiles — against the project's proposed architecture without affecting the production workspace.

It is not a one-time migration tool — it manages the coexistence period and the eventual cutover for each domain.

8.2 Nightly Audit Job
A scheduled Lambda function runs nightly to detect drift between the DynamoDB registry, the DSL repository, and the Ardoq workspaces.
Orphan detection: Components present in Ardoq EAMS workspaces but not in DynamoDB registry. Indicates a component was created in Ardoq directly (bypass of integration layer) or a registry write failed.
Ghost detection: Registry entries pointing to Ardoq IDs that no longer exist in Ardoq. Indicates a component was deleted from Ardoq manually.
Staleness detection: Registry entries whose last_compiled_at is more than 30 days older than the corresponding DSL file's last Git commit. Indicates a compile failure left the entity unsynced.
Field drift detection: Samples 5% of EAMS components per domain, compares Ardoq field values against current DSL content. Detects direct Ardoq field edits bypassing the integration layer.

ADR-INT-001: Five reference types, not fifteen
Decision: Compress 15 DSL relationship types to 5 Ardoq reference types, carrying semantic type as a reference attribute.
Context: Ardoq Viewpoints require explicit configuration for each reference type they include. The more reference types exist, the more complex each Viewpoint definition becomes. With 15 types, every Viewpoint update requires managing a combinatorial matrix of type inclusions.
Consequence: Viewpoints filter on eams_rel_type attribute values rather than Ardoq reference types. This shifts filtering logic from the Ardoq Viewpoint definition to the EAMS Viewpoint Engine query layer, where it is easier to maintain. The tradeoff is that Ardoq's native UI shows all integration relationships the same color unless a field-based color rule is configured.
Alternative rejected: One Ardoq reference type per DSL type. Rejected due to Viewpoint complexity and the ongoing maintenance burden of adding new Ardoq reference types when the metamodel is extended.

Diff-Aware Write Decision Logic (per entity)
function shouldWrite(eamsId, compiledHash):
entry = registry.lookup_by_eams_id(eamsId)

if entry == null:
# Entity is new — not yet in Ardoq
return { action: CREATE, ardoqId: null }

if entry.content_hash != compiledHash:
# Entity exists but has changed — content hash mismatch
return { action: UPDATE, ardoqId: entry.ardoq_id }

# Entity exists and is unchanged — skip write
return { action: SKIP, ardoqId: entry.ardoq_id }

Pass | Entities Written | Rationale
1 | Workspaces (if new domains present) | Workspaces must exist before any components can be written to them.
2 | Shared entities: Goals, Standards, NFR Profiles, Actors | Cross-cutting entities in eams-shared workspace. Domain entities reference these — they must exist first.
3 | Systems (C4 L1), Capabilities, Processes, Data Entities, Data Products, Infrastructure Nodes, Decisions | Top-level domain entities with no intra-domain parent dependencies.
4 | Containers (C4 L2), Interfaces | Require parent System to exist. Containers are children of Systems in Ardoq hierarchy.
5 | Components (C4 L3) | Require parent Container to exist.
6 | All relationships (all types) | All source and target components are guaranteed to exist after passes 1–5.
7 | Retirements / deletions | Processed last to avoid breaking references that are being simultaneously updated.
Force-full-sync rate limit risk
Writing thousands of components in a single pipeline run will approach or exceed Ardoq's API rate limits.
Force-full-sync must use the batched write strategy (Section 5.5) with conservative concurrency settings.
For full-tenant syncs (all domains), force-full-sync should be run domain by domain, not all at once.
Schedule force-full-sync runs outside of business hours when possible.

Physical deletion policy
Physical deletion from Ardoq is a manual, privileged operation performed by the EAMS platform engineer after a domain architect review.
A component may be physically deleted only after it has been retired for a minimum of 12 months and has no active inbound references.
Physical deletion requires removing the registry entry from DynamoDB and the DSL file from Git history is preserved.
Batched Write Pseudocode
function batchWrite(changeSet, options):
const { concurrency = 5, batchSize = 50, retryMax = 3 } = options

# Group changes by pass (dependency order)
const passes = groupByWritePass(changeSet)

for (pass of passes):
const batches = chunk(pass.changes, batchSize)

for (batch of batches):  // Process batches sequentially within a pass
await Promise.allSettled(  // Concurrent within batch
batch.map(change => writeWithRetry(change, retryMax))
)
await sleep(200)  // 200ms between batches — rate limit headroom

# Collect results
return { written, skipped, failed, alerts }

Parameter | Default | Force-Full-Sync | Rationale
concurrency | 5 | 3 | Concurrent writes within a batch. Reduced for force-full-sync to leave headroom for other API consumers.
batchSize | 50 | 25 | Components per batch. Ardoq processes component writes synchronously — smaller batches for full sync avoids timeout.
inter-batch delay | 200ms | 500ms | Pause between batches. Increased for full-sync to reduce sustained API pressure.
retry max | 3 | 3 | Retry attempts on 429 (rate limit) or 5xx responses.
retry backoff | 1s/2s/4s | 2s/4s/8s | Exponential backoff. Longer for full-sync where rate limit recovery is slower.

Scenario Lifecycle
# 1. BRANCH CREATED (first compile of proposal/experiment branch)
POST /api/v1/scenario
{ "name": "EAMS — [proposal] Claims RT Adjudication", "mainWorkspace": "claims-workspace-id" }
# → Store scenario_id in DynamoDB workspace entry for branch

# 2. SUBSEQUENT COMPILES (incremental changes to the branch)
POST /api/v1/scenario/{scenario_id}/component  // Add new component to scenario
PATCH/api/v1/scenario/{scenario_id}/component/{component_id}  // Modify existing
DELETE/api/v1/scenario/{scenario_id}/component/{component_id}  // Mark for deletion in scenario

# 3. BRANCH MERGED → main  (after review approval)
#    The merge to main triggers a main-branch compile — changes are
#    written to the production workspace by the standard upsert engine.
#    The Scenario is then deleted.
DELETE /api/v1/scenario/{scenario_id}

# 4. BRANCH ABANDONED / REJECTED
DELETE /api/v1/scenario/{scenario_id}
# → Remove workspace entry from DynamoDB

Scenario naming convention
Format: EAMS — [{intent}] {branch.name from branch.yaml}
Examples:
  EAMS — [proposal] Claims Real-Time Adjudication Initiative
  EAMS — [experiment] Explore Event Mesh for Claims
The EAMS prefix ensures EAMS-managed Scenarios are visually distinguishable from manually created Scenarios in the Ardoq UI.
Project Branch Workspace Lifecycle
# 1. PROJECT BRANCH CREATED (first compile)
#    Clone the current main workspace state to a new workspace
POST /api/v1/workspace/{main_workspace_id}/copy
{ "name": "eams-claims-branch-rt-adjudication" }
# → Register new workspace in DynamoDB with workspace_type: branch-copy
# → Provision same component types and reference types as main workspace

# 2. COMPILE BRANCH CONTENT
#    Force-full-sync the branch DSL content to the copy workspace
#    (All entities in scope of .eams/branch.yaml are written)
#    Entities NOT in the branch scope are inherited from the workspace copy

# 3. SUBSEQUENT COMMITS — diff-aware writes to the copy workspace
#    Uses a branch-specific DynamoDB partition to track copy workspace IDs

# 4. DIFF REPORT — generated against main workspace
#    Comparison: branch copy workspace vs main workspace
#    Output: structured diff report (added / modified / removed entities and relationships)
#    Posted to Git pull request as a rendered Markdown summary

# 5. APPROVED AND MERGED TO MAIN
#    Main branch compile runs — writes approved changes to main workspace
#    Branch copy workspace is retained for 30 days (audit trail), then deleted
DELETE /api/v1/workspace/{branch_workspace_id}  // After 30-day retention
# → Remove workspace entry and all branch-partition entries from DynamoDB

ADR-INT-002: Full workspace copy for project branches
Decision: Project branches (intent: project) receive a full Ardoq workspace copy, not a Scenario.
Context: Scenarios cannot support the full Ardoq Viewpoint experience. A project branch represents architecture that will be implemented — reviewers need to run Viewpoints, assess NFR profiles, and validate integration surfaces with the same tooling they use for production architecture.
Consequence: Branch workspace copies consume Ardoq workspace quota. Long-lived project branches with many domains in scope will create multiple workspace copies. Mitigation: scope branches narrowly (per .eams/branch.yaml scope block) and enforce branch lifetime limits.
Alternative rejected: Scenarios for all branch types. Rejected because Scenario limitations (no full Viewpoint support, no cross-workspace scenario references) would make project branch review significantly less rigorous than main branch review.
Workspace | Content | Managed By | Write Access
eams-{domain}` (new) | EAMS-authored entities. Growing as architects create DSL files. | EAMS compiler pipeline | EAMS compiler only (service account)
eams-legacy-{domain}` (existing) | Pre-EAMS entities from TPM sync and manual entry. Read-only during migration period. | Frozen — TPM sync disabled for domain on migration start | Read-only (all users)
Why freeze the legacy workspace rather than continuing the TPM sync?
Allowing the legacy TPM sync to continue writing to the legacy workspace during migration creates a moving target — architects are trying to migrate content that keeps changing.
Freezing the legacy workspace gives a stable baseline to assess, migrate from, and eventually deprecate.
The Salesforce TPM → EAMS sync is re-established after domain migration, routing through the EAMS integration layer rather than the legacy direct sync.

Category | Examples | Pipeline Behavior | Recovery
Hard failure
(compile blocked) | Schema validation failure, referential integrity failure, ID stability violation, Ardoq workspace not found | Pipeline fails immediately. No writes attempted. Error report posted to PR. | Fix DSL files or workspace config. Re-push to trigger pipeline.
Soft failure
(compile continues) | Standards compliance finding, NFR regression warning, review date overdue | Pipeline continues. Findings written to compliance report artifact. PR comment includes summary. | Architect acknowledges findings. Exception ADR created if applicable.
Write failure
(partial compile) | Ardoq API 4xx/5xx on specific component write, rate limit hit after retries exhausted | Upsert engine continues with remaining entities. Failed writes collected in failure report. DynamoDB not updated for failed entities. | Re-run pipeline (idempotent). Failed entities will be retried on next compile. If persistent, check Ardoq status and API credentials.
Registry drift
(Ardoq diverged from DSL) | Component deleted from Ardoq manually, Ardoq field edited directly, registry entry corrupt | Detected by nightly audit job. Alert posted to EAMS ops channel. | Run eams compile --force-full-sync --domain {domain} for affected domain.
Workspace missing | Ardoq workspace deleted, workspace ID changed after Ardoq migration | Hard failure on any compile for that domain. | Run eams workspace provision --domain {domain} then eams compile --force-full-sync.
Direct Ardoq editing policy
Architects must not edit components in EAMS-managed Ardoq workspaces directly via the Ardoq UI.
All architecture data changes must go through the DSL → compile pipeline.
The nightly audit job will detect and alert on direct edits. Repeated violations are escalated to the EAMS governance process.
The only permitted direct Ardoq operations are: read-only browsing, Viewpoint configuration, and Scenario management for EAMS-managed Scenarios.
