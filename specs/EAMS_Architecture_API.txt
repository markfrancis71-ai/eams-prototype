1. Overview
The EAMS Architecture API is the read layer of the EAMS ecosystem. It exposes the compiled architecture graph — maintained in OpenSearch and Ardoq — to all consumers: the Architect Workbench, the diagram renderer, the Confluence artifact generator, the LLM advisory layer, the signal enrichment pipeline, and future external integrations. No consumer interacts with Ardoq or OpenSearch directly.
The API exposes two complementary surfaces that share a single underlying query layer:
1.1 Read Strategy: Hybrid OpenSearch + Ardoq

1.3 Architectural Position
The API is strictly read-only. The write path flows exclusively through the DSL compiler pipeline → Ardoq Integration Layer. The API enforces this at the infrastructure level: API Gateway and AppSync are configured with no mutation resolvers and no POST endpoints that accept entity data.

2.2 Role Definitions
2.3 Service Token Scoping
Service tokens are issued with an explicit endpoint_scope claim that restricts which endpoint groups the token may call. This limits blast radius if a service credential is compromised.
2.4 Authorization Enforcement
Authorization is enforced at the API Gateway and AppSync authorizer layer before any resolver executes. The following rules apply to all endpoints:
Requests without a valid JWT or service token receive 401 Unauthorized.
Requests with a valid token but insufficient role or scope receive 403 Forbidden.
Service tokens attempting to call endpoints outside their scope receive 403 Forbidden with error code SCOPE_INSUFFICIENT.
Reader role requests to endpoints that return sensitive fields (data_classification, compliance_status, eams_architect email) have those fields redacted from the response. The entity is returned with redacted fields set to null and a redacted: true flag.
All authorization decisions are logged to CloudWatch with token identity, endpoint, and decision. Failed authorization generates a CloudWatch metric alarm after 10 failures per minute per token.
3. REST API — Endpoint Reference
All REST endpoints are versioned under /v1/. The base URL is https://api.eams.{company}.com/v1. Responses are JSON. All list endpoints support pagination via limit and cursor query parameters (default limit: 50, max: 200).

Searches across entity names, descriptions, and indexed attributes using OpenSearch full-text search. Supports faceted filtering by domain, kind, status, criticality, lifecycle, data_classification, and standard compliance. Returns ranked results with match highlights.
3.5 Branch & Diff Endpoints
Returns all active non-main branches registered in the EAMS system, with their branch intent, review status, Ardoq representation type, and scope.
Returns the diff report generated by the last compile of the specified branch. The diff describes all entity additions, modifications, removals, relationship changes, standards violations introduced, and NFR regressions relative to the base branch.

Single query layer
Both the GraphQL and REST surfaces are projections of the same underlying query layer — an OpenSearch-backed graph query engine that resolves entity graphs from the materialized architecture index.
Neither surface has independent data access logic. Adding a new REST endpoint or a new GraphQL field does not require changes to data access — only to the projection layer.
Query Type | Data Source | Rationale
Entity lookup by ID or domain | OpenSearch | Sub-10ms response. Satisfies signal pipeline <200ms SLA.
Full-text and faceted search | OpenSearch | Native capability. Powers architecture discovery in Workbench.
Relationship traversal (1–3 hops) | OpenSearch | Pre-indexed relationship edges. Fast for the majority of viewpoint queries.
Deep graph traversal (4+ hops) | Ardoq REST API passthrough | Ardoq's graph engine handles arbitrary-depth traversal more efficiently than OpenSearch join chains.
Branch/Scenario comparison | Ardoq REST API passthrough | Branch diffs require comparing two Ardoq graph states — Ardoq Scenario API is purpose-built for this.
Viewpoint resolution | OpenSearch (query) + Viewpoint Engine (logic) | Viewpoint Engine applies filter and projection logic on top of OpenSearch graph query results.
Provenance / compile history | DynamoDB ID Registry (read) | Git SHA, compiled_at, and migration source are in the registry, not OpenSearch.

ADR-API-001: API is read-only by design
Decision: The Architecture API exposes no write operations for entity or relationship data.
Context: All architecture data changes must flow through the DSL → Git → compile pipeline to maintain version history, referential integrity, and standards compliance. An API write path would bypass all of these controls.
Consequence: Clients that need to modify architecture data (e.g. the Workbench editor) generate DSL files and submit them via Git, not via the API. The API may expose a /draft endpoint in a future version that accepts DSL snippets and returns validation results without committing — but this is advisory only, not a write path.
Exception: The /feedback endpoint (Section 6) accepts signal data from the signal enrichment pipeline. This is operational telemetry, not architecture data, and does not modify the entity graph.

GraphQL vs REST — when to use which
Use GraphQL when: you need multiple nested entity types in a single request (Workbench entity panel, diagram subgraph fetch), you need to specify exactly which fields to return, or you need subscription-based updates (Workbench real-time refresh).
Use REST when: you need stable URLs for caching or sharing, you are building an external integration with a predictable access pattern, or you are using a non-JavaScript client without a mature GraphQL library.

Denormalised relationship index
The outbound_rel_index and inbound_rel_index arrays on each entity document are the key performance mechanism for 1–2 hop traversal queries.
For a 1-hop traversal, the API performs a single OpenSearch GetById, reads outbound_rel_index, and performs a multi-get for the target entity IDs. This avoids a graph join query entirely.
For 2-hop traversal, the API fetches the 1-hop targets, reads their outbound_rel_index arrays, and performs a second multi-get. Still no graph join.
For 3+ hops, the API falls through to Ardoq passthrough (Section 1.1).

Branch endpoint caching
Endpoints called with a non-main branch parameter are never cached at the API Gateway layer.
Branch data changes on every commit — caching branch responses would cause architects to see stale branch state in the Workbench.
The AppSync resolver for branch queries bypasses the response cache and always reads from the branch-specific OpenSearch index.
