1. Overview
The EAMS Viewpoint Engine is the layer that transforms the raw architecture graph into diagrams, rendered outputs, and stakeholder-appropriate views. It sits between the Architecture API's query layer and the rendering targets (Structurizr, Mermaid, D2, Confluence). Given a viewpoint definition and a scope, the engine resolves the applicable entities and relationships, applies styling and layout hints, and produces a canonical render graph that projects to one or more output formats.
The engine has four responsibilities:
1.1 Design Decisions
Full query expressiveness: Viewpoint scope expressions support attribute filters and relationship traversal rules, not just domain/kind scoping. This enables security, compliance, and impact viewpoints that require conditional graph traversal.
Both static and parameterised viewpoints: Static viewpoints produce one diagram for a fixed scope (e.g. enterprise capability map). Parameterised viewpoints are templates instantiated with a scope argument (e.g. C4 container view for system.claims-processing). Both are expressed in the same definition language.
Split authoring ownership: A standard library of viewpoints is maintained by the EAMS platform engineer. Domain architects can author custom viewpoints in their domain DSL directory, subject to the same Git review process as entity changes.
Layout hints, not pinned layout: Viewpoint definitions carry layout hints (direction, grouping, rank) that are advisory to the renderer. Stable, predictable diagrams without requiring a coordinate persistence layer. Pinned layout is a future phase capability.
Structurizr DSL as primary renderer: C4 is the architectural modeling language. Structurizr DSL is C4's native format. All C4 viewpoints use Structurizr as the primary renderer. Mermaid is the secondary renderer for Confluence embedding. D2 handles non-C4 viewpoints.
Canonical render graph as intermediate: All format projectors share a single canonical intermediate representation. Adding a new output format requires only a new projector — no changes to resolution logic.

5.3 Traverse Rule Execution
Traverse rules follow relationship edges from a named result set. For depth ≤ 2, the engine uses the outbound_rel_index and inbound_rel_index arrays pre-indexed on each OpenSearch entity document (described in the Architecture API specification). After all include rules have been evaluated, the engine resolves the relationship set. It fetches all relationships where both the from_id and to_id are in the entity set, using the relationship index.

The Render Graph Builder takes the resolved entity and relationship sets, applies the viewpoint's styling rules and layout hints, and assembles the canonical render graph — a format-agnostic intermediate representation that all format projectors consume. The canonical render graph is also the full_json response format returned by the API.

7. Format Projectors
Format projectors translate the canonical render graph to a target output format. Each projector is a pure function: render_graph → string. Projectors do not query the architecture graph — all data they need is in the canonical render graph.
7.1 Structurizr DSL Projector
The Structurizr DSL projector is the primary projector for C4 viewpoints. It produces a valid Structurizr DSL workspace definition that can be rendered by the Structurizr CLI, Structurizr Lite, or the Structurizr cloud service. The projector maps C4 view types from the viewpoint definition's c4_view_type field.
7.2 Mermaid Projector
The Mermaid projector produces Mermaid diagram syntax for embedding in Confluence, GitHub, and Notion. It maps to flowchart for most viewpoints, classDiagram for data model viewpoints, and sequenceDiagram for process flow viewpoints. Mermaid does not natively support C4 component boundaries, so groups are rendered as subgraphs.
7.3 D2 Projector
The D2 projector is the primary output for non-C4 viewpoints: capability maps, blast radius diagrams (radial layout), domain interaction maps, and process flow diagrams. D2's flexible layout engine and container support make it well-suited to these use cases.

8. Custom Viewpoint Authoring Guide
Domain architects can author custom viewpoints for their domain. Custom viewpoints follow the same VDL syntax as the standard library. They live in .eams/viewpoints/{domain}/ and go through the same Git review process as entity changes.
8.1 Authoring Workflow
Identify the stakeholder concern the viewpoint addresses. A viewpoint without a clear stakeholder is usually not needed.
Determine whether the viewpoint is static (one fixed diagram) or parameterised (instantiated per entity).
Check the standard library first — a custom viewpoint may not be needed if a standard viewpoint covers the use case with a different scope parameter.
Create the .viewpoint.yaml file in .eams/viewpoints/{domain}/.
Validate locally: eams viewpoint validate --file path/to/viewpoint.yaml
Test resolution: eams viewpoint resolve --id viewpoint.my-viewpoint --scope-system system.claims-processing --dry-run. Inspect the resolved entity and relationship counts.
Preview output: eams viewpoint preview --id viewpoint.my-viewpoint --scope-system system.claims-processing --format structurizr_dsl
Commit and submit for review. Pull request includes the rendered preview as a diagram artifact.

9. Staleness & Refresh Model
Viewpoint renders are point-in-time snapshots of the architecture graph. The graph changes on every compile. The staleness model defines how the Workbench knows when a rendered diagram is out of date and how it communicates that to the architect.
9.1 Staleness Detection
Every canonical render graph carries a compiled_at timestamp and a git_sha for the compile that produced it. Staleness is detected via the Architecture API's entityUpdated GraphQL subscription, which fires whenever the compile pipeline writes entity changes to OpenSearch.
9.2 Refresh Behaviour
Manual refresh: Architect clicks refresh on the diagram. The Workbench calls GET /viewpoints/{id}/resolve with the same parameters. The new render graph replaces the current one. If the diagram changed, a brief change summary is shown (N entities added, M relationships changed).
Auto-refresh: Disabled by default. Architects can enable auto-refresh per diagram in the Workbench. Auto-refresh fires on the entityUpdated subscription event, subject to a 30-second debounce to prevent rapid sequential refreshes during a bulk compile.
Stale age warning: If a rendered diagram has not been refreshed and its git_sha is more than 5 commits behind the current main branch HEAD, the Workbench shows an elevated stale warning with the number of commits behind.

kind: viewpoint

name: C4 Container View
description: Shows the internal containers of a software system, their technologies, and their interactions. C4 Level 2.
version: 1.2  # Viewpoint definition version — increment on breaking changes
metadata:
author: platform-engineering@company.com
last_reviewed: 2026-02-01
tags:
- c4
- technical

# ═══ STAKEHOLDERS ════════════════════════════════════════════════════
# Who this viewpoint is designed for. Informational — used for catalogue filtering.
stakeholders:
- developer  # primary
- architect  # primary
- business  # secondary

# ═══ PARAMETERS ══════════════════════════════════════════════════════
# Parameterised viewpoints declare their scope inputs here.
# Static viewpoints omit this section entirely.
parameters:
- name: scope_system  type: entity_id  kind: system  required: true
description: The system whose containers will be rendered.
- name: branch  type: string  required: false  default: main

# ═══ SCOPE ═══════════════════════════════════════════════════════════
# Defines the seed entity set. This is the starting point for resolution.
scope:
strategy: parameter-seed  # parameter-seed | domain-query | entity-query
# parameter-seed: seed from a declared parameter value
# domain-query:   seed from a domain + kind filter
# entity-query:   seed from an explicit attribute filter expression
seed:
from_parameter: scope_system  # Use the scope_system parameter as the seed entity

# ═══ RESOLUTION RULES ════════════════════════════════════════════════
# What to include relative to the seed. Evaluated in declared order.
include:

# Rule 1: Include the seed entity itself
- rule: seed_entity  label: The system being viewed

# Rule 2: Include all containers belonging to the seed system
- rule: related_entities
relationship_types: [container_of]  # synthetic rel: system→its containers via Ardoq parent hierarchy
direction: children          # children: traverse into Ardoq component hierarchy
entity_kinds: [container]
label: Containers of the system

# Rule 3: Include external systems that containers interact with
- rule: traverse
from: containers             # 'containers' = entities matched by rule 2
relationship_types: [uses, consumes, publishes-to, subscribes-to]
direction: outbound
entity_kinds: [system, container]
depth:
label: External dependencies

# Rule 4: Include actors that interact with the seed system at boundary
- rule: traverse
from: seed  relationship_types: [uses, invokes]
direction: inbound  entity_kinds: [actor]  depth:
label: Users and actors

# ═══ FILTERS ═════════════════════════════════════════════════════════
# Applied after resolution. Entities not passing filters are excluded.
filters:
entity_filters:
# Exclude retired entities
- field: status  operator: not_in  values: [retired]
relationship_filters:
# Only show relationships between entities that are IN the resolved set
- rule: internal_only  # Exclude rels where one endpoint is not in the entity set

# ═══ STYLING ═════════════════════════════════════════════════════════
# Display rules applied to nodes and edges in the render graph.
styling:
node_styles:
# Style by entity kind
- match: {kind: system, role: seed}  # The seed system
shape: box  color: #1B3A5C  font_color: #FFFFFF  border: bold

- match: {kind: container}      # All containers

parameter-seed | Seeds resolution from the value of a declared parameter. The parameter value is an entity ID. | Parameterised viewpoints that scope to a specific system, container, or capability. | C4 Container View, Component View, Capability Detail
domain-query | Seeds resolution from all entities in a domain matching kind and status filters. | Domain-scoped static viewpoints showing all entities of a type within a domain. | Domain System Landscape, Domain Capability Map

Rule Type | Description | Key Fields | Notes
seed_entity | Includes the seed entity or entities themselves in the result set. | — | Always the first rule. Without it the seed does not appear in the diagram.
related_entities | Includes entities connected to the current result set via a specific relationship type or Ardoq hierarchy (parent/children). | relationship_types, direction, entity_kinds | Use direction: children to traverse Ardoq component hierarchy (system→containers).
traverse | Traverses relationship edges from a named result set to a specified depth, following specified relationship types. | from (named set), relationship_types, direction, depth, entity_kinds | Named sets: 'seed', 'containers', 'capabilities', or any rule label value. depth max: 5.
attribute_filter | Adds entities from the full graph matching an attribute expression, independent of the current result set. | field, operator, values | Used to pull in related entities by attribute rather than by graph edge. E.g. 'include all standards governing any entity in result set'.
union | Combines multiple named result sets into one. Useful for assembling complex scopes from simpler parts. | sets (list of rule labels) | Result is the union of all named sets. Duplicates are deduplicated by eams_id.
Operator | Applies To | Description | Example
eq | Any field | Exact equality. | field: status, operator: eq, value: active
not_eq | Any field | Exact inequality. | field: lifecycle, operator: not_eq, value: retired

not_in | Any field | Value is not in a list. | field: status, operator: not_in, values: [retired]
contains | String fields | Field value contains substring. | field: name, operator: contains, value: Claims

gt` / `lt | Numeric fields | Greater / less than. | field: nfr_profile.availability_target, operator: gt, value: 99.9

Reserved Name | Refers To

1 | Parameter binding — resolve parameter values to entity IDs | DynamoDB ID Registry (fast lookup by eams_id) | Bound parameter map: { scope_system: 'system.claims-processing' }
2 | Scope strategy execution — produce the seed entity set | OpenSearch eams-entities index | Seed entity set (one or more entity documents)
3 | Include rule evaluation — expand the result set rule by rule | OpenSearch (depth ≤ 2), Ardoq passthrough (depth 3–5) | Accumulated entity set, updated after each rule
4 | Entity filter application — remove excluded entities | In-memory filter over accumulated set | Filtered entity set
5 | Relationship resolution — fetch all relationships between entities in the set | OpenSearch eams-relationships index (lookup by from_id/to_id) | Relationship set connecting entities in the result
6 | Relationship filter application — remove excluded relationships | In-memory filter over relationship set | Filtered relationship set
7 | External entity enrichment — fetch display fields for traversal targets not yet fully loaded | OpenSearch multi-get | Complete entity documents for all entities in result set

Traverse rule — OpenSearch multi-hop execution (depth 2)
# Step 1: Read outbound_rel_index from all entities in the 'from' set
#         Filter to matching relationship_types and criticality
const targetIds = fromSet
.flatMap(e => e.outbound_rel_index)
.filter(rel => matchingRelTypes.includes(rel.rel_type))
.filter(rel => passesRelationshipFilters(rel))
.map(rel => rel.to_id)
.filter(id => !alreadyInResultSet(id))

# Step 2: Multi-get entity documents for all target IDs
const newEntities = await opensearch.mget({
index: "eams-entities", ids: targetIds
})

# Step 3: If depth == 2, repeat from Step 1 with newEntities as the 'from' set
#         If depth > 2, switch to Ardoq passthrough
if (remainingDepth > 0 && depthSoFar <= 2) {
executeTraverseRule(newEntities, remainingDepth - 1)
} else if (remainingDepth > 0) {
return ardoqPassthrough(newEntities, remainingDepth, filters)
}

Relationship resolution and the internal_only filter
The query above returns all relationships where BOTH endpoints are in the resolved entity set. This is what the internal_only relationship filter does implicitly.
Viewpoints without internal_only can show 'dangling' relationships to entities outside the viewpoint scope — valid for some views (e.g. integration surface) but confusing for others (e.g. C4 container view).
The internal_only filter is recommended as the default for all C4 viewpoints and any viewpoint where the boundary of the diagram is semantically meaningful.

Custom viewpoint governance
Custom viewpoints are scoped to their authoring domain by default — they cannot query entities in other domains without explicit cross-domain scope declaration.
Custom viewpoints with depth > 3 traverse rules require platform engineer review before merge.
Custom viewpoints must not duplicate an existing standard library viewpoint with only a scope change — use the standard viewpoint with a scope parameter instead.
Viewpoints with entity-query scope strategy that return > 500 seed entities require a performance review annotation in the viewpoint file.
The total number of custom viewpoints per domain is soft-capped at 20. Beyond this, platform engineer review is required to assess whether consolidation is needed.
Event | Staleness Impact | Workbench Behaviour
Entity in the render graph is modified | Render is stale | Workbench displays stale indicator on diagram. Offers one-click refresh.
New entity is added to a domain in scope | Render may be stale | Workbench offers refresh if new entity would be included by the viewpoint's scope expression. Evaluated client-side against scope metadata.
Relationship between two entities in the render graph changes | Render is stale | Stale indicator shown. Refresh re-runs full resolution.
Compile completes with no changes to entities in the render graph | Render is current | No indicator. compiled_at and git_sha updated silently.
Branch compile completes | Branch render is stale | Branch-specific renders display stale indicator for that branch.
Staleness does not mean incorrectness
A stale diagram is one that was accurate as of a previous compile. It may still be entirely accurate — if the changes since the last compile do not affect any entity in the diagram's scope.
The Workbench communicates staleness as informational, not as an error. Architects working in a fast-moving team may prefer to refresh diagrams deliberately rather than continuously.
