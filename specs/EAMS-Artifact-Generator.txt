1. Overview
The EAMS Artifact Generator transforms the compiled architecture graph into publishable artifacts for consumption beyond the Workbench. It produces four artifact types: Confluence domain pages (auto-generated, always-current architecture documentation), dynamic HTML reports (self-contained interactive architecture summaries), viewpoint diagram bundles (packaged diagram exports in multiple formats), and change reports (what-changed summaries derived from compile diffs). These artifacts carry the architecture from the authoritative graph to the audiences — business stakeholders, development teams, governance boards, and external reviewers — who need it in a form suited to their context.
The Artifact Generator is a Lambda-based pipeline invoked by two triggers. Confluence domain pages are auto-published after every successful main branch compile — they stay current without architect intervention. All other artifact types are on-demand: an architect triggers them explicitly via the eams artifact CLI command or from the Workbench Artifact panel. This split keeps routine documentation automatic while keeping intentional, point-in-time artifacts under architect control.
1.1 Design Decisions
Automatic trigger for Confluence domain pages only: Confluence domain pages are living documentation — they should reflect the current compiled graph without requiring manual action. Every successful main branch compile triggers a Confluence page refresh for any domain with changed entities. All other artifact types are on-demand: dynamic HTML, diagram bundles, and change reports are intentional artifacts produced for a specific purpose and audience.
Dynamic HTML as a first-class artifact type: Dynamic HTML reports are self-contained single-file artifacts that can be distributed via email or a file share without requiring Confluence access or the Workbench. They embed the entity data as JSON and render interactively in the browser — architects can navigate entities, filter by status or criticality, and expand relationship detail without a network connection. This makes them suitable for external distribution (auditors, external partners, review boards) and for point-in-time snapshots.
Confluence pages are generated, not authored: Confluence domain pages are fully generated from the architecture graph on every compile. They are not editable by Confluence users — attempting to edit them is futile because the next compile overwrites changes. A prominent banner on every generated page states this. Confluence is a distribution channel, not an authoring surface.
Diagram bundles use the canonical render graph: Diagram bundles package diagrams in all supported formats (Structurizr DSL, Mermaid, D2, SVG, PNG) from the canonical render graph. The bundle format is a zip archive with a manifest, making it suitable for import into other tools and for archival purposes.
Change reports are diff-driven: Change reports are generated from the compile diff record — the added, modified, and removed entity sets computed during compilation. They do not require a separate diff computation step. A change report can be generated for any pair of compile records on any branch.
LLM-assisted prose for Confluence and HTML: The natural language explanation output from the LLM Advisory Layer is optionally embedded in Confluence domain pages and HTML reports as architecture prose sections. When present, these sections are clearly labelled as AI-generated and are subject to the same acknowledgement gate as other advisory outputs.

2. Confluence Domain Pages
Confluence domain pages are the primary external documentation surface for the EAMS ecosystem. One page is generated per domain per successful main branch compile (if the domain has any changed entities). Pages are structured, consistently formatted, and always current — they reflect the compiled graph state as of the triggering compile.

2.3 Confluence Publishing Model
Each domain maps to a single Confluence page in a configured Confluence space. The page ID is stored in the Artifact Generator's DynamoDB table after first creation and reused on subsequent updates. If the page is deleted from Confluence, the next compile creates a new page and updates the stored page ID.
3. Dynamic HTML Reports
Dynamic HTML reports are self-contained single-file artifacts. All entity data, relationship data, and diagram definitions are embedded as JSON in the HTML file. The browser renders the report interactively — no server, no Workbench, no Confluence access required. The file can be emailed, placed on a file share, or committed to a repository.

A viewpoint diagram bundle is a zip archive containing a set of diagrams in all supported formats for a specified scope. Bundles are used for distribution to teams who need diagrams for presentations, documentation tools, or architecture tooling that imports specific formats.

4.2 SVG and PNG Generation
SVG files are generated by rendering the canonical render graph JSON using a headless Chromium instance (via Puppeteer) running in a Lambda layer. The D3.js rendering code from the Workbench is reused — the headless browser loads a minimal HTML harness, renders the diagram, and extracts the SVG element. PNG files are generated by rasterising the SVG at 2x resolution.
5. Change Reports
Change reports summarise what changed in the architecture graph between two compile records. They are the primary artifact for communicating architectural changes to stakeholders at release time, during branch reviews, or as part of governance reporting.

5.3 LLM Change Narrative
The change narrative is an optional LLM-generated section that provides architectural context for the changes — explaining the intent, the impact, and the significance of what changed in language a business or executive audience can understand. It is distinct from the structural diff sections, which are purely data-driven.
6. Pipeline Architecture
The Artifact Generator is a Lambda-based pipeline. Auto-triggered artifacts (Confluence domain pages) are invoked by an SQS event published by the Compiler at the end of a successful main branch compile. On-demand artifacts are invoked via the Architecture API's /v1/artifacts/generate endpoint, which enqueues a generation job and returns a job ID for polling.

Shell | Hand-crafted HTML/CSS | No framework dependency — must work offline and in restricted browser environments. Tailwind CSS compiled and inlined.

Navigation | Table of contents with anchor links | Auto-generated from report sections. Fixed sidebar on wide viewports, collapsible on narrow.
No external requests from the HTML report
The generated HTML file makes no external network requests. All assets (CSS, JS, diagrams) are inlined. This is a firm requirement: the report must be usable in air-gapped environments and must not leak architecture data to third-party CDNs.
The one exception is the optional Mermaid.js CDN path for environments where file size matters. The default is inlined Mermaid. The CDN option is explicitly configured in the artifact generator config and displays a warning banner when active.

Headless rendering adds latency
SVG/PNG generation via headless Chromium adds 2–5 seconds per diagram. For a domain bundle with 10 diagrams, this is 20–50 seconds of additional generation time. Bundles with SVG/PNG are generated asynchronously — the CLI command returns immediately with a bundle job ID and the architect polls or receives a notification when the bundle is ready.
Bundles with source-format files only (Structurizr DSL, Mermaid, D2) are generated synchronously and are typically ready in under 5 seconds.

# System prompt excerpt:
You are an enterprise architect explaining architectural changes to a business audience.
You have access to a structured diff of architecture changes:
- Added entities: systems, containers, capabilities, or decisions that are new
- Modified entities: existing entities with field or relationship changes
- Removed entities: entities that have been retired or removed

Write 2-4 paragraphs that explain:
1. What architectural intent these changes represent (not just what changed)
2. Which business capabilities or domains are affected
3. Any significant risks or dependencies introduced
4. What remains unchanged and stable

Ground every claim in the provided diff data. Do not invent changes not present in the data.
Respond ONLY with the prose narrative. No headers. No bullet points. No markdown.

Stage | Name | Description | Failure Behaviour
1 | Job initialisation | Read job parameters. Validate scope entity/domain exists. Acquire compile record for the specified branch and SHA. Resolve applicable viewpoints for scope. | Job fails with INVALID_SCOPE. No external writes.
2 | Data assembly | Fetch entity records, relationship records, compliance findings, decisions from Architecture API context bundle. Assemble into generation context. | Job fails with CONTEXT_UNAVAILABLE. No external writes.
3 | LLM prose generation | If LLM prose requested and an acknowledged advisory explanation exists: fetch from advisory record. If not: optionally request new explanation and wait (on-demand only). Auto-publish jobs skip LLM prose if no acknowledged advisory exists. | LLM failure: skip prose section, continue with data-only artifact. Log gap.
4 | Diagram resolution | Call Architecture API viewpoint resolve endpoint for each applicable viewpoint. Collect canonical render graphs. | Individual viewpoint failure: skip that diagram, continue. Log skipped viewpoints.
5 | Artifact rendering | Render the artifact (Confluence markup, HTML, ZIP bundle). For SVG/PNG: invoke headless Chromium Lambda layer. | Rendering failure: job fails with RENDER_ERROR. No partial publish.
6 | Publishing | Publish to target: Confluence API (page create/update), S3 presigned URL (HTML/ZIP), or return inline (small artifacts). | Confluence API failure: retry twice with exponential backoff. On third failure: job marked failed with PUBLISH_ERROR, compile not rolled back.
7 | Job record finalisation | Write completed job record to DynamoDB. Publish artifact-ready event. Notify architect (on-demand) or log (auto). | Always runs even on failure — ensures job record reflects final state.

# SNS -> SQS -> Artifact Generator Lambda
# Lambda spawns one generation job per changed domain:
for (const domain of changed_domains) {
enqueueJob({
artifact_type: 'confluence_domain_page'
scope_domain: domain
compile_id: event.compile_id
trigger: 'auto'
})
}

Auto-publish parallelism
When a compile changes multiple domains, the Artifact Generator spawns one Lambda invocation per changed domain simultaneously (up to 5 concurrent — configurable). This means the total auto-publish time is bounded by the slowest single domain, not the sum of all domains.
If all 5 domains change in a single compile and each takes 30s, the total auto-publish time is 30s, not 150s.
